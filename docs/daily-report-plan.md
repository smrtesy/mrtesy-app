# דוח יומי — תוכנית תכנון (כלי יום חמישי)

מסמך תכנון, יולי 2026. כלי יום חדש במסגרת "כלי היום" של smrtTask
(`docs/day-tools-plan.md`). סטטוס: **טיוטה לאישור המשתמש** — אין לכתוב קוד
או להריץ מיגרציות לפני אישור.

הכלי מאפשר למשתמש להגדיר בעצמו **פריטים** (שאלות) לדיווח יומי, כשלכל פריט
יש **אפשרויות תשובה** משלו ולכל תשובה **ניקוד אופציונלי**. המשתמש עונה על
השאלות פעם ביום (צ'ק-אין קצר), והמערכת מייצרת **אוטומטית** דוח מצטבר
שבועי עם ניקוד כללי וספירת תשובות לכל שאלה. הדוח מגיע **כל יום שלישי
ל"נכנס" (inbox) של smrtTask** — כפריט שממנו אפשר לצפות בדוח המלא.

---

## 1. העיקרון

- הכלי יושב במסגרת כלי-היום הקיימת: מתג בהגדרות + קונפיג משלו
  (`user_settings.day_tools`, לפי `docs/day-tools-plan.md §2.1`). כבוי =
  המערכת מתנהגת כאילו לא קיים, בלי למחוק דאטה שנכתב.
- **הגדרה גמישה על ידי המשתמש**: לא סכימה קשיחה בקוד. המשתמש בונה את
  רשימת השאלות, האפשרויות והניקוד ממסך ההגדרות. שינוי ההגדרות לא דורש
  מיגרציה או פריסה.
- **המילוי היומי ידני, הדוח אוטומטי**: השאלות ("התפללת במנין?",
  "למדתי חת״ת?") הן דיווח-עצמי שרק המשתמש יכול לענות עליו, ולכן המילוי
  היומי ידני. ה"אוטומטי" מתייחס לייצור הדוח — צבירה, חישוב ניקוד וספירת
  תשובות — שקורה לבד לפי לוח זמנים, וגם בלחיצת כפתור.
- **כל התאריכים/השעות בשעון ניו יורק** (`America/New_York`) — גבולות היום
  לצורך "יום דיווח", טווחי התקופה, ושעת הייצור המתוזמנת. אחסון ב-UTC,
  המרה לניו יורק רק בגבול התצוגה/החישוב (לפי `CLAUDE.md`).

## 2. מודל הנתונים

שלוש ישויות: **הגדרות** (שאלות + אפשרויות), **דיווחים יומיים** (התשובות),
ו**ריצות דוח** (תמונת מצב מצטברת לתקופה).

### 2.1 הגדרת הפריטים

```
פריט (item)  = שאלה אחת
  label       — טקסט השאלה ("התפללת במנין?")
  position    — סדר תצוגה
  active       — פעיל/ארכיון (לא מוחקים כדי לא לאבד היסטוריה)

אפשרות (option) = תשובה אפשרית לפריט
  label       — טקסט התשובה ("כן" / "לא" / "חצי")
  score       — ניקוד, אופציונלי (100 / 0 / 50 — או ריק)
  position    — סדר תצוגה
```

- **מספר אפשרויות חופשי** לכל שאלה (לא רק כן/לא/חצי).
- **הניקוד אופציונלי**: פריט ללא ניקוד באף אפשרות → נספר בלבד (טאלי), לא
  נכנס לחישוב הניקוד הכללי. פריט עם ניקוד → נכנס גם לניקוד.

### 2.2 הדיווח היומי

רשומה אחת לכל (משתמש, תאריך, פריט): איזו אפשרות נבחרה באותו יום.

```
דיווח (entry)
  entry_date    — תאריך היום בשעון ניו יורק
  item_id       — לאיזו שאלה
  option_id     — איזו תשובה נבחרה
  score_snapshot — הניקוד של האפשרות ברגע המענה (מוקפא!)
```

> **למה `score_snapshot`**: אם המשתמש ישנה בעתיד את הניקוד של "כן" מ-100
> ל-80, הדוחות ההיסטוריים חייבים להישאר כפי שהיו. הניקוד מוקפא ברגע
> המענה, בדיוק כמו ש-`daily_plans` מקפיאה את ה-picks לפני שה-rollover מנקה.

### 2.3 ריצת דוח (תמונת מצב)

כשנוצר דוח (מתוזמן או ידני) שומרים תמונת מצב, כך שהדוח יציב וניתן לצפייה
חוזרת:

```
ריצה (run)
  period_type   — 'weekly' | 'monthly'
  range_start / range_end — גבולות התקופה (בשעון ניו יורק, נשמר כתאריך)
  overall_score — הניקוד הכללי המחושב (null אם אין ניקוד כלל)
  breakdown     — JSONB: לכל שאלה, ספירת כל תשובה + ניקוד ממוצע לשאלה
  generated_by  — 'schedule' | 'manual'
  created_at
```

## 3. חישוב הדוח

לתקופה הנבחרת (טווח תאריכים בשעון ניו יורק):

1. **ספירת תשובות לכל שאלה** — כמה פעמים נבחרה כל אפשרות בטווח. זהו הפלט
   המרכזי שביקשת:
   ```
   למדתי חת״ת
     כן  – 2
     לא  – 3
     חצי – 7
   ```
2. **ניקוד לכל שאלה** (אם לשאלה יש ניקוד) — ממוצע ה-`score_snapshot` של
   הדיווחים שלה בטווח.
3. **ניקוד כללי** — מוצג רק אם קיים ניקוד כלשהו. **ממוצע** כל
   ה-`score_snapshot` בטווח על פני כל השאלות המנוקדות, מוצג כמספר בסקאלה
   שהמשתמש בחר (למשל 0–100). *(החלטת המשתמש: ממוצע.)*
4. **סעיף משימות אוטומטי** (לפי הבקשה "דוח שיכלול משימות יומיות") — מושך
   מהמערכת הקיימת בלי דיווח ידני, לתקופה:
   - **משימות שנסגרו לפי גודל** — מהיר / בינוני / גדול (ספירה), מתוך
     `tasks.completed_at` בטווח (או `daily_plans.completed_*`).
   - **זמן עבודה** — סכום `work_sessions.worked_seconds` בטווח.
   סעיף זה מוצג לצד שאלות הדיווח-העצמי. (שלב 2 — ראו §7.)

## 4. הגדרות (UI)

הרחבת `DayToolsSettings.tsx` — כשהכלי `dailyreport` פעיל נפתח קונפיג
(מתקפל, לפי עקרון ה-UI הקומפקטי ב-`CLAUDE.md`):

- **עורך שאלות** — רשימת שאלות; לכל שאלה טקסט + רשימת אפשרויות (טקסט +
  שדה ניקוד אופציונלי) + הוספה/מחיקה/סידור. ברירת מחדל: מקופל, נפתח
  בלחיצה.
- **תקופת הדוח** — שבועי (ברירת מחדל; חודשי כאופציה עתידית).
- **תזמון** — הדוח השבועי מגיע כל **יום שלישי** ל"נכנס" של smrtTask.
  ניתן להגדיר את השעה (שעון ניו יורק) ולכבות את הייצור האוטומטי.
- הכול נשמר דרך `api()` → `PATCH /api/me/settings` (מיזוג ברמת כלי). ראו
  הערה ב-§6 לגבי מה נשמר ב-JSONB מול טבלאות.

## 5. איסוף יומי (הצ'ק-אין)

*(החלטת המשתמש: לא כפתור — משימה יומית בראש המהיר.)*

- כל יום מופיעה **משימה "מלא דוח יומי" בראש רשימת המשימות המהירות**
  (`size = quick`, מוצמדת לראש). לחיצה עליה פותחת טופס קצר עם שאלות היום;
  בחירת אפשרות אחת לכל שאלה ושמירה. השלמת המשימה = הדיווח מולא.
- **מנגנון**: המשימה נוצרת/מוצגת פעם ביום (גבול היום בשעון ניו יורק). היא
  אינה משימת inbox רגילה — היא נגזרת ממצב הדיווח של היום (חסר דיווח →
  מופיעה; מולא → נעלמת/מסומנת). ההצמדה לראש דרך נקודת העגינה "מבנה
  השולחן" של `TaskList` (`docs/day-tools-plan.md §2.4`) — לא רשומת `tasks`
  רגילה שתיכנס ל-rollover ולספירות המשימות. *(הפרדה חשובה: משימת הדיווח
  לא נספרת בסעיף "משימות שנסגרו" של הדוח.)*
- אם הכלי כבוי → המשימה לא מופיעה כלל.

## 6. אחסון ו-API

- **קונפיג הכלי** (`enabled`, `period_type`, `schedule_*`) → ב-
  `user_settings.day_tools.dailyreport` (JSONB), כמו שאר כלי-היום. הוספת
  `dailyreport` ל-`DayToolSlug` וב-`DAY_TOOLS` (`src/lib/smrttask/day-tools.ts`).
- **שאלות/אפשרויות/דיווחים/ריצות** → טבלאות ייעודיות (תקדים: method131→
  `daily_plans`, workclock→`work_sessions`; כלי-יום עם היסטוריה מקבלים
  טבלה משלהם). מיגרציות חדשות תחת `supabase/migrations/`:
  - `daily_report_items` (+ `daily_report_options`)
  - `daily_report_entries` (unique על user_id+entry_date+item_id)
  - `daily_report_runs`
  - כולן `user_id`+`org_id`, RLS "בעלים בלבד" (תקדים `daily_plans`).
- **API (Express, org-scoped, `requireApp("smrttask")`)**:
  - `GET/PUT /api/daily-report/items` — קריאה/עריכה של השאלות.
  - `GET/PUT /api/daily-report/entries?date=` — התשובות של יום.
  - `POST /api/daily-report/generate` — ייצור ידני ("צור דוח עכשיו").
  - `GET /api/daily-report/runs` — היסטוריית דוחות + צפייה.
- **ג'וב מתוזמן** — endpoint מוגן ב-secret משותף (`x-cron-secret`, תקדים
  `sync.ts` / `smrtbot/jobs.ts:334`) שרץ שעתי, ומייצר דוח למי שהגיע זמנו:
  **יום שלישי** בשעה שנקבעה בשעון ניו יורק, פעם בשבוע (once-per-week
  guard, תקדים `daily_summary_last`). מיגרציית pg_cron לפי
  `20260617150000_smrtbot_daily_summary_cron.sql`. **קריטי**: ייבוא
  edge/deno דרך `npm:`/`jsr:` בלבד, לא `esm.sh` (CLAUDE.md).
- **משלוח ל"נכנס"** — הייצור יוצר גם פריט ב"נכנס" של smrtTask שמפנה לדוח
  (תקדים: פריטי ה"הצעה" של סשני Claude ב-`claude-session.ts`, ו-
  `task_type`/`action_links` הקיימים). הפריט נושא קישור עמוק לדוח המלא.

## 7. שלבים

1. **תשתית + הגדרה + מילוי**: רג'יסטרי, טבלאות, API של items/entries, עורך
   השאלות בהגדרות, וטופס הצ'ק-אין היומי. i18n (he+en) בכל קומיט.
2. **דוח**: חישוב הצבירה (ספירות + ניקוד), `runs`, מסך צפייה בדוח, כפתור
   "צור עכשיו", וסעיף המשימות האוטומטי.
3. **תזמון**: ג'וב שעתי + מיגרציית pg_cron + once-per-period guard.
4. **(עתידי)** משלוח הדוח בוואטסאפ/מייל; תקופות נוספות (רבעוני/שנתי);
   גרפים/מגמות.

## 8. החלטות שנסגרו

כל השאלות הפתוחות אושרו — אין חוסמים לפני קוד:

1. **תדירות ומשלוח** — דוח **שבועי**, מגיע **כל יום שלישי ל"נכנס"** של
   smrtTask (לא וואטסאפ/מייל בשלב זה). חודשי = אופציה עתידית.
2. **ניקוד כללי** — **ממוצע** ה-`score_snapshot` בטווח.
3. **סעיף המשימות האוטומטי** — משימות שנסגרו לפי גודל (מהיר/בינוני/גדול)
   + זמן עבודה. נכלל בשלב 2.
4. **מילוי יומי** — **משימה "מלא דוח יומי" בראש המהיר**, לא כפתור (§5).
