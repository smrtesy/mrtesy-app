smrtTask — הצינור המלא של ייבוא וסיווג

מסמך אימות לאחר ענף claude/refine-task-classification-B5vQI. כל מספר, כל קונפיגורציה, וכל קובץ — מאומת מקריאה בקוד או מהנתונים החיים בפרודקשן.

איך לקרוא את המסמך
המסמך מסודר לפי הזרימה הלוגית: מקור הודעה → איסוף → סיווג מקדים → סיווג AI → יצירת משימה → תצוגה → לחיצת כפתור. בכל סעיף ציינתי מאיזה קובץ ומאיזו שורה הנתון מגיע. בסוף יש סעיף "ההגדרות בקיצור" עם כל הקנובים שניתן להזיז.

קבועי החלון
calendar_past_days ו-calendar_future_days נטענים מטבלת smrttask_system_params (סופר-מנהל). ברירת המחדל לשניהם היא 1. ניתן לערוך ב-/admin/apps/smrttask/parameters.
איך נקבע כלל 8 (קטגוריות Gmail) פר-משתמש
ai-process קורא מ-rules_memory את כל השורות עם trigger='category=<key>' (4 ערכים: promotions / social / forums / updates). אם יש שורה — היא מנצחת. אם לא קיימת — חל ברירת המחדל החכמה: promotions, social, forums מסוננים. updates עוברים לסונט. המשתמש שולט בזה דרך הדף /settings/smrttask/rules תחת "Gmail Filters" (4 toggles).
תוויות Gmail רק להודעות מ-v6 ואילך
gmail-sync v6 (הקומיט הזה) הוא הראשון ששומר metadata.labels. הודעות שנשמרו לפני זה (178 הודעות gmail נכון לעכשיו) לא מכילות labels — אצלן כלל 8 לעולם לא מופעל, וההודעה תיפול ל-needs_claude ותעובד דרך Haiku ו-Sonnet כרגיל. רק הודעות gmail חדשות שייכנסו אחרי 19/5 ייהנו מסינון הקטגוריות.

6. שלב 3 — זיהוי המשך: threadId ל-Gmail, source_message_id ל-WhatsApp

חידוש שהוסף בקומיט 28bb2ff ל-Gmail, והורחב ב-d85b2be ל-WhatsApp, כדי לפתור את בעיית הכפילויות (46% מהמשימות בעבר). supabase/functions/ai-process/index.ts:372-427

רק אחרי ש-Haiku סיווג את ההודעה כ-ACTIONABLE, ai-process בודק שני מסלולים לפי source_type:

// ── מסלול WhatsApp (whatsapp / whatsapp_echo) ───────────────────── // whatsapp-webhook עושה upsert על שורה אחת ב-source_messages לכל שיחה // ומאפס processing_status='pending' בכל הודעה חדשה. בלי dedup, היינו // יוצרים משימה חדשה לכל תשובה. הפתרון: לחפש משימה פתוחה שכבר קושרה // לאותה source_message_id. if (source_type === "whatsapp" || source_type === "whatsapp_echo") { openTask = tasks WHERE user_id = userId AND status IN ('inbox', 'in_progress') AND source_message_id = msg.id ORDER BY created_at DESC LIMIT 1; // נמצאה → append עדכון, לא משימה חדשה. אחרת → ממשיך לשלב 4. } // ── מסלול Gmail (השרשור הקלאסי) ─────────────────────────────────── if (source_type === "gmail" && msg.metadata.threadId) { siblings = source_messages WHERE user_id = userId AND source_type = 'gmail' AND metadata->>'threadId' = msg.metadata.threadId AND id != msg.id; openTask = tasks WHERE user_id = userId AND status IN ('inbox', 'in_progress') AND source_message_id IN siblings ORDER BY created_at DESC LIMIT 1; // נמצאה → append עדכון, חוסך קריאה ל-Sonnet. אחרת → ממשיך לשלב 4. }

אם נמצאה משימה לקישור — ההודעה נכנסת כעדכון ל-tasks.updates, רשומת task_activities נוצרת עם activity_type='thread_followup', וה-classification מסומנת actionable_followup ב-log_entries. אחרת — ממשיך לשלב 4.

8. שלב 5 — יצירת המשימה (Sonnet)

הקריאה היקרה ביותר בכל הצינור. supabase/functions/ai-process/index.ts:237-323

הפרומפט המלא ל-Sonnet

לחץ להצגת הפרומפט המלא (74 שורות) 
You are a task builder for a personal task system.
Extract concrete actionable tasks from this message.
Return ONLY a JSON Array, no markdown, no commentary.

═══ EMPTY-ARRAY RULE ═══
Return [] (empty array) when the message is purely informational:
  • Marketing / newsletter / sale / promotion
  • Bank/payment confirmation of an already-completed transaction
  • System receipts already handled by the recipient
  • Status updates that need no human action
The caller will record an empty result as informational.

═══ TASK SHAPE ═══
{
  "title_he":     "Hebrew, starts with action verb",
  "description":  "Hebrew, 2-3 sentences: WHAT / WHO / WHEN / consequences",
  "priority":     "urgent|high|medium|low",
  "reason_he":    "Why this task and why this priority — cite ONE concrete fact",
  "due_date":     "YYYY-MM-DD or null",
  "ai_actions": [
    { "label":  "3-7 Hebrew words naming recipient or next step",
      "prompt": "Full instruction for the AI to run, in English or Hebrew" }
  ],
  "owner_contact": "name + phone + email or null"
}

═══ TITLE RULES (mandatory) ═══
Verb-first only: לענות / לאשר / להחליט / להעביר / לבדוק / להתקשר /
לפגוש / לתאם / להזמין / להגיש / להכין / לדחות / לבטל / לחתום / לשלם.

BAD:  "תיאום פגישה"     (noun, not a command)
BAD:  "מייל מ-X"         (passive)
GOOD: "לתאם פגישת קליטה עם Amalgamated Bank עד 25/5"
GOOD: "לאשר לדינה את הזמן (שני 09:00 או רביעי 15:00)"

═══ PRIORITY RULES (mandatory) ═══
urgent : deadline today/tomorrow AND a concrete fact (amount, named
         person, blocked system).
high   : deadline within 7 days AND impacts people other than the user.
medium : deadline within 30 days OR routine follow-up.
low    : no clear deadline OR soft/optional action OR upcoming auto-renewal.

Never default to urgent. If you can't cite a concrete urgency fact, drop
to medium.

Auto-system notifications (Vercel, Railway, GitHub, monitoring services)
→ max medium, unless production is currently down.

═══ CONTENT-SPECIFIC RULES ═══
1. Subscription renewal notice ("your X plan renews on Y for $Z"):
   priority: "low". description MUST list, in this order:
     • מה מתחדש (service + plan)
     • כמה ייחויב (amount + currency)
     • מתי (date)
     • איך לבטל / לשנות (link or step from the message)
   ai_actions should include "draft cancel" or "review subscription".

2. Bank / payment confirmation of a completed transaction → return [].

═══ AI_ACTIONS RULES ═══
2-3 actions per task. The label is the button text the user sees — it
MUST name the recipient or the concrete next step, not the generic
action name. The prompt is what the AI will run on click; include enough
context that the AI doesn't need to re-read this message.

קלט

שדות: From, To, Subject, גוף ההודעה דרך bodyForAI(msg) — ל-Gmail/Drive/Calendar זה body_text, ל-WhatsApp זה raw_content שמכיל 20 הודעות אחרונות של השיחה כדי לתת לסונט את ההקשר המלא. כל זה חתוך ל-body_truncate_task תווים (ברירת מחדל 6000). מוסיף גם הקשר פרויקט אם נמצא.

פלט וטיפול

אם Sonnet החזיר []ההודעה מסווגת בחזרה ל-informational — לא נוצרת משימה
אם Sonnet החזיר משימה אחת או יותרכל אחת נשמרת ב-tasks עם status='inbox', manually_verified=false
אם Sonnet החזיר טקסט שלא ניתן לפרסנוצרת משימה אחת עם הטקסט הגולמי כתיאור, priority='medium', והערה ב-reason_he
שדות משימה שנשמריםtitle_he, description, priority, due_date, ai_actions, related_contact, ai_confidence=0.8, ai_model_used
תיעודשורה ב-log_entries עם category='ai_process', ה-reason_he של סונט (במקום ה-reason של היייקו), עלות כוללת ב-USD, וטוקנים

נקודה 3 — Calendar: 0 שורות. מה השורש

הפעלתי שאילתה אבחונית מקיפה:

בדיקהתוצאהפרשנות
user_credentials WHERE service='google_calendar'2 שורות, יש refresh_token + scopes=['calendar']נראה תקין
user_settings WHERE calendar_connected=true2 משתמשיםשני המשתמשים סימנו "מחובר"
sync_state WHERE source='google_calendar'0 שורותאף פעם לא נשמר nextSyncToken
source_messages WHERE source_type='google_calendar'0 שורותאף webhook לא הגיע
log_entries WHERE category='calendar_webhook'0 שורותGoogle אף פעם לא קרא ל-webhook
log_entries WHERE category='calendar_renew_watch'6 כשלונות מאז 25/04, כולם "Token refresh failed: 401"השורש

שורש הבעיה: calendar-renew-watch רץ כל 6 שעות, ובכל הפעלה זה נכשל ב-401 כשהוא מנסה לרענן את ה-Calendar access token דרך oauth2.googleapis.com/token עם ה-refresh_token של שירות google_calendar. בלי access token תקין הוא לא מצליח להירשם ל-watch של Google, ולכן Google אף פעם לא קורא ל-calendar-webhook, ולכן אין שורות source_messages.

מעניין במיוחד: ה-refresh_token של service='gmail' של אותם משתמשים עובד מצוין (Gmail מסונכרן כל 2 דקות מ-25/04). הבעיה ספציפית ל-refresh_token של google_calendar. הסיבות האפשריות:

  1. ה-OAuth grant של Calendar התקבל מ-client_id אחר ממה שה-Edge Function מחזיק ב-GOOGLE_CLIENT_ID env var. כשרה client לא מתאים לרענון הוא נכשל ב-401.
  2. המשתמש ביטל הרשאה ל-Calendar ב-Google Account (לא תקפ ל-Gmail).
  3. ה-refresh_token של Calendar נשמר מ-flow ישן/שונה ומאז סובב.
הפתרון
  1. למשתמש: להתנתק ולהתחבר מחדש ל-Google Calendar דרך דף ה-Connections. ה-OAuth flow המעודכן יביא refresh_token חדש שתואם ל-GOOGLE_CLIENT_ID הנוכחי.
  2. למפתח: לוודא שכל ה-Google services (Gmail/Drive/ Calendar) מתחברים ב-OAuth flow אחד שמבקש את כל ה-scopes בבת אחת, ולא בשלושה flows נפרדים שעלולים להחזיק tokens מ-clients שונים.
  3. אבחון מהיר: אחרי שמשתמש מתחבר מחדש, להריץ ידנית את calendar-renew-watch (POST ל-Edge Function עם x-cron-secret). אם זה מצליח, יש שורה חדשה ב-sync_state ו-Google יתחיל לשלוח webhooks תוך דקות.
מסמך זה משקף את מצב הקוד בענף claude/refine-task-classification-B5vQI אחרי איחוד ה-sync ל-Edge Functions ומחיקת part1-collector.ts. גרסאות פרוסות: ai-process v18, gmail-sync v6, whatsapp v10 (פרוס אבל לא מקבל traffic), calendar-webhook v3 (לא נקרא — ראה נקודה 3). כל הצהרה במסמך מקושרת לקובץ ושורה. אם משהו אינו תואם — זו תקלה במסמך, לא בקוד.
