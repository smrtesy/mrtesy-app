# כלי היום — תוכנית מסגרת "ניצול אפקטיבי של היום"

מסמך תכנון, יולי 2026. משפחת כלי-עזר שיושבים **מעל** מערכת ההצעות/המשימות
של smrtTask, כשכל כלי הוא תוסף (add-on) שהמשתמש מפעיל או מכבה בהגדרות.
הכלי הראשון: **מהיר·3·1** (כל המהירות + 3 בינוניות + 1 גדולה ליום).

מסמך זה משלים את `tasks-unification-spec.md` (מפרט שולחן העבודה) ולא
מחליף אותו. סטטוס: טיוטה לאישור המשתמש.

---

## 1. העיקרון

- **שכבת המשימות נשארת ניטרלית.** משימות, גדלים, תאריכים, נכנס, בריכה —
  קיימים תמיד. כלי-יום הוא *שיטת עבודה* שמולבשת מעליהם: תצוגה, מכסות,
  טקסים, ופונקציות מילוי.
- **כל כלי = מתג בהגדרות** + קונפיגורציה קטנה משלו. כבוי = המערכת מתנהגת
  כאילו הכלי לא קיים (בלי למחוק דאטה שהכלי כתב).
- **הכלים לא מתנגשים** — כל כלי מוגדר מול "נקודות עגינה" קבועות (ראו §2.4),
  כך שאפשר להפעיל כמה כלים במקביל.
- עיצוב: לפי עקרון ה-UI הקומפקטי — כל כלי מוסיף לכל היותר כפתור-אייקון
  אחד שקט למסך; שום סרגל קבוע.

## 2. מסגרת התוספים (התשתית המשותפת)

### 2.1 אחסון — עמודת JSONB אחת

מיגרציה אחת: `user_settings.day_tools jsonb NOT NULL DEFAULT '{}'`.

```json
{
  "method131": { "enabled": true, "medium_quota": 3, "big_quota": 1 },
  "marathon":  { "enabled": true }
}
```

> **החלטה (יולי 2026):** המרתון הוא כלי במסגרת עם Switch בהגדרות
> (ברירת מחדל: פעיל). כשכבוי — כפתור "התחל ריצה" בכותרת המהיר לא מוצג,
> ומסך המרתון לא נגיש. הריצות ההיסטוריות ב-`marathon_runs` נשמרות.

- מפתח = slug של הכלי; ערך = `{ enabled, ...config }`.
- למה JSONB ולא עמודה בוליאנית פר-כלי (התקדים של `show_ai_costs`):
  צפויים 5–8 כלים, לכל אחד קונפיג משלו; JSONB חוסך מיגרציה לכל כלי חדש
  ושומר את כל "שיטת היום" של המשתמש באובייקט אחד.
- חשיפה: הוספת `day_tools` ל-`UPDATABLE_SETTINGS` ב-
  `server/src/modules/platform/me/routes.ts` (מיזוג עמוק ברמת כלי —
  PATCH של כלי אחד לא דורס את השאר).

### 2.2 רג'יסטרי בצד הלקוח

`src/lib/smrttask/day-tools.ts` — מקור אמת אחד:

```ts
export const DAY_TOOLS = [
  { slug: "method131", icon: ListChecks, defaultConfig: { medium_quota: 3, big_quota: 1 } },
  { slug: "marathon",  icon: Zap,        defaultConfig: {} },
] as const;
```

- כותרת/תיאור ב-i18n: `dayTools.<slug>.title` / `.desc` (he+en באותו קומיט).
- hook: `useDayTool("method131")` → `{ enabled, config }` (ברירות מחדל
  מהרג'יסטרי כשאין רשומה). נטען פעם אחת מ-`GET /api/me/settings` דרך
  provider בעמוד המשימות.

### 2.3 מסך הגדרות — "כלי היום"

סקשן חדש בהגדרות smrtTask (תחת `settings/apps/smrttask`): רשימת כלים,
לכל אחד Switch + שורת תיאור, וקונפיג שנפתח רק כשהכלי פעיל (מתקפל כברירת
מחדל). שמירה דרך `api()` → `PATCH /api/me/settings` — **לא** דרך קליינט
Supabase ישיר (בניגוד לעמוד הפרמטרים הקיים, שהוא חוב טכני).

### 2.4 נקודות עגינה (איפה כלי יכול "לשבת")

כדי שכל כלי עתידי יתחבר בלי לפרק את TaskList מחדש:

| עגינה | מיקום | דוגמה |
|---|---|---|
| מבנה השולחן | חלוקת הרשימות + מכסות ב-`TaskList` | מהיר·3·1 |
| כפתור פעולה בכותרת רשימה | אייקון שקט ליד כותרת | מרתון ("התחל ריצה"), "בנה את היום" |
| טקס בוקר | ה-hook של `MorningStart` (פעם ביום) | אשף בניית יום |
| טקס ערב | באנר/מסך בסוף יום (חדש) | סיכום ערב |
| צינור ה-AI | שדות בפרומפט ה-classifier / קריאת הצעה ייעודית | הצעת יום, שיוך גודל |
| התראות | מנגנון ה-notifications הקיים | תזכורת "לא בנית יום" |

### 2.5 אכיפה בצד השרת

כלי עם endpoint משלו (למשל "בנה לי יום") בודק את הטוגל וקורא
`day_tools` מהשרת — לא סומך על הלקוח. כלים שהם תצוגה בלבד נאכפים
בלקוח בלבד.

## 3. כלי 1 — מהיר·3·1 (השלמה)

**השיטה:** כל יום מסיימים את *כל* המהירות, ובוחרים במודע 3 בינוניות
ו-1 גדולה. כל השאר בבריכה.

### 3.1 מה כבר קיים (בנוי, עובד)

- `tasks.size` — quick/medium/big (מיגרציה `20260707180000`).
- `planned_for` + rollover לילי לפי אזור-זמן (`20260707190000`, `20260708210000`).
- חלוקת השולחן: מהיר–עכשיו / נבחרו-להיום / בריכה + מכסה רכה 3+1
  (`TaskList.tsx`), גרירה, כפתור +/− לבחירה להיום.
- מרתון מהיר מלא (`MarathonMode.tsx` + `marathon_runs`).
- ביקורת ותק (`ReviewBanner`).

### 3.2 מה חסר כדי שזה יהיה "תוסף שלם"

1. **מיתוג (gating).** כל התנהגות 3·1 ב-`TaskList` עוברת מאחורי
   `useDayTool("method131")`.
2. **מצב כבוי מוגדר.** כשהכלי כבוי חוזרים לכלל מהמפרט המקורי
   (`tasks-unification-spec.md` §1): מהיר | רגיל, עלייה אוטומטית לשולחן
   לפי "יעד ≤ 3 ימי עסקים ולא חסומה", והבריכה מוצגת כ"ממתינות" ממוינת
   לפי תאריך. בלי צ'יפים של מכסה, בלי כפתור בחירה-להיום. `planned_for`
   לא נמחק — פשוט לא משפיע על התצוגה.
3. **מכסות קונפיגורביליות.** `medium_quota` (ברירת מחדל 3) ו-`big_quota`
   (1) מהקונפיג; הצ'יפים `desk.mediumQuota`/`desk.bigQuota` קוראים מהם.
4. **פונקציית ההכלה — באנר "בנה את היום" + מבנה 4 הרשימות** (§3.3) —
   זה הלב החסר.
5. **גודל מה-AI (מהיר/בינוני)** (§3.4) — בלי זה רשימת המהיר לא אמינה.
6. ניקוי: מפתח ה-i18n הישן `sizeQuick: "⚡ מהיר (הצעת AI)"` שמשקר כרגע.

### 3.3 "בנה את היום" — באנר בחירה + מבנה ארבע הרשימות

**החלטת עיצוב (יולי 2026):** הבחירה **ידנית** דרך **באנר בראש דף
המשימות** — לא אשף/חלון קופץ. בשלב זה אין הצעת-AI לבחירה (נדחה — §3.6).

**המבנה:**

- כל ההצעות (אחרי אישור) וכל המשימות יושבות ב**רשימה כללית אחת** —
  המאגר. אין הפרדת "שולחן/בריכה" מוקדמת; ההפרדה נוצרת מהבחירה היומית.
- **מדי יום, בכניסה הראשונה לדף המשימות** (מנגנון localStorage יומי,
  כמו `MorningInboxRedirect`), עולה **באנר "בנה את היום"** בראש הדף
  (אותו דפוס של `ReviewBanner`/`UpcomingBanner` הקיימים — שקט, נסגר).
- הבאנר מזמין לבחור **3 בינוניות + 1 גדולה**. המהירות **אינן נבחרות** —
  כולן נכנסות אוטומטית (עושים את כולן). בחירה = הקשה על מועמדת
  (planned_for=today + today_position). המכסה נשארת **רכה** — אפשר לחרוג
  עם אזהרה עדינה.
- אחרי הבחירה (או סגירת הבאנר), הדף מציג **ארבע רשימות, מלמעלה למטה:**
  1. **מהיר** — כל המשימות המהירות. ("התחל ריצה" של המרתון יושב כאן.)
  2. **בינוני** — שלוש שנבחרו להיום.
  3. **גדול** — האחת שנבחרה להיום.
  4. **השאר** — כל מה שלא נבחר (בינוני/גדול שלא נכנסו להיום), רשימה
     אחת מתחת, ממוינת לפי דדליין אפקטיבי → נפילה מאתמול → ותק → עדיפות.
     (יורשת את תפקיד "הבריכה" הנוכחית, מוצגת שטוחה במקום מקובצת.)
- הבאנר נשאר זמין דרך כפתור-אייקון שקט בכותרת (☀️) לפתיחה חוזרת באמצע
  היום, וקישור "נבנה את היום" בסוף טריאז' הבוקר בנכנס.

> שינוי מהטיוטה הקודמת: זו רשימה כללית + באנר + 4 רשימות תצוגה, במקום
> אשף-חריצים. פשוט יותר ותואם את עקרון ה-UI הקומפקטי.

**היסטוריה וסטטיסטיקה — `daily_plans` (טבלה חדשה):**

`(id, user_id, org_id, plan_date, picked_task_ids uuid[], quick_total,
completed_quick, completed_medium, completed_big, closed_at)` + RLS.
נכתבת כשבונים יום ומתעדכנת ב-rollover. נותנת: רצף ימים ("בנית יום
7 ימים ברצף"), אחוז השלמה, ובסיס לטקס-ערב עתידי. בלי הטבלה אין דרך
לשחזר מה נבחר אתמול (rollover מוחק `planned_for`).

### 3.4 המסווג קובע מהיר/בינוני — לא "גדול" (החלטה, יולי 2026)

ה-classifier (`server/src/prompts/classifier.ts`) לא פולט `size` בכלל —
כל משימה חדשה נולדת `medium`. השלמה:

- הוספת שדה `size` לסכמת ה-JSON של בונה-המשימות, **בשני ערכים בלבד:**
  **מהיר** = פעולה אחת תחומה בלי הכנה (לענות/לאשר/להתקשר/לקבוע/לשלוח);
  **בינוני** = כל השאר. **בספק → בינוני** (שומר על אמינות רשימת המהיר).
  זהו בדיוק אותו שיקול שהמסווג כבר עושה כשהוא מסמן משימה ל"עכשיו".
- **"גדול" הוא החלטה אנושית בלבד.** ה-AI לעולם לא מסמן משימה כגדולה —
  זו בחירת המוקד היומית המודעת של המשתמש (ה-1 מתוך "3·1"). משימה הופכת
  לגדולה רק דרך הטאגל הידני / הבחירה בבאנר.
- זה שיפור דאטה גלובלי (לא תלוי-טוגל) — הגודל שימושי גם בלי הכלי.
- תיקוני משתמש (⚡↔בינוני↔גדול) כבר נאספים דרך reorder/toggle; בהמשך
  יוזנו כחוקים ל-rules_memory אם יתגלו דפוסים (כמו במפרט המקורי).

### 3.5 מה נשאר רך בכוונה

- אין נעילת מכסה קשיחה. אין חסימת בחירה רביעית.
- מרתון ממשיך לעבוד גם על "נבחרו להיום" (קיים היום) — לא נוגעים.

### 3.6 נדחה לשלב מאוחר יותר

- **הצעת-AI ליום** ("הצע לי יום" — כפתור ✨ שבוחר את ה-3+1 עם נימוק,
  והמשתמש מאשר/מחליף). בשלב זה הבחירה ידנית לגמרי. כשייבנה:
  `POST /api/tasks/day-plan/suggest`, קריאת Haiku אחת, שום `planned_for`
  לא נכתב בלי אישור, ושמירת קישורים עמוקים כלשונם בפרומפט.
- אין שינוי בסטטוסים ובזרימת הנכנס.

## 5. שלבי ביצוע (מהיר·3·1)

1. **תשתית המסגרת:** מיגרציה `day_tools` · whitelist ב-`/me/settings`
   (מיזוג עמוק) · רג'יסטרי + hook · סקשן "כלי היום" בהגדרות · העברת
   המרתון לרג'יסטרי (טוגל, ברירת מחדל פעיל).
2. **מיתוג 3·1:** gating של המכסות/בחירה-להיום ב-`TaskList` · מצב כבוי
   לפי כלל 3-ימי-עסקים · מכסות מהקונפיג · מבנה 4 הרשימות (מהיר/בינוני/
   גדול/השאר) · ניקוי i18n.
3. **בנה את היום:** באנר הבחירה בראש הדף (ידני) · `daily_plans` +
   כתיבה/עדכון · כניסות בוקר/כפתור-☀️/אחרי-טריאז'.
4. **AI (גודל):** `size` בפרומפט ה-classifier — **מהיר/בינוני בלבד**
   (גדול = אנושי). שיפור דאטה גלובלי, לא תלוי-טוגל.
5. **ליטוש:** רצף ימים על הבאנר · חיווי בסוף מרתון כשכל היום הושלם.

> הצעת-AI ליום (§3.6) אינה בשלבים אלה — נדחתה.

כל שלב עומד בפני עצמו וניתן לדחיפה נפרדת (כולל פרוטוקול ה-pre-push המלא).

## 6. שינויי דאטה (סיכום)

- `user_settings.day_tools jsonb NOT NULL DEFAULT '{}'`
- טבלת `daily_plans` + RLS + אינדקס `(user_id, plan_date)` ייחודי
- עדכון פונקציית ה-rollover: לפני איפוס `planned_for` — לקבע את מצב
  ההשלמה ב-`daily_plans` של אתמול
- אין שינוי ב-`tasks` (הכול כבר קיים)

## 7. החלטות שנסגרו (יולי 2026)

- **מרתון** = טוגל במסגרת, ברירת מחדל פעיל (§2.1).
- **מצב כבוי של מהיר·3·1** = חזרה לכלל "יעד ≤ 3 ימי עסקים עולה
  לשולחן" (אפשרות א', §3.2).
- **בחירת היום** = ידנית, דרך באנר בראש הדף + מבנה 4 רשימות (§3.3).
  הצעת-AI ליום נדחתה (§3.6).
- **המסווג** קובע **מהיר/בינוני בלבד**; "גדול" הוא החלטה אנושית (§3.4).
- **ברירת מחדל לכלי** = פעיל למשתמשים קיימים (שקוף, ה-UI הקיים), נבחר
  באונבורדינג לחדשים.

## 8. כלי — פוקוס תוכנית (בלוק זמן יומי על תוכנית smrtPlan) + בונה-תוכנית AI

**הרעיון:** פרויקט ארוך יושב כתוכנית ב-smrtPlan. המשתמש מתחייב ל**כמות
זמן יומית קבועה** ומריץ מדי יום **סשן פוקוס** מתוזמן. הטיימר מציג את
**השלב הנוכחי** (המשימה המוכנה הבאה שלו בתוכנית), ובסוף הזמן — צליל +
מסך חוסם. סיום שלב תוך כדי הסשן משחרר את הבא בשרשרת — כולל לאיש צוות.

זו **שכבת ביצוע** מעל smrtPlan, לא מבנה נתונים חדש למשימות. רוב הצנרת
כבר קיימת.

### 8.1 למה smrtPlan ולא checklist על המשימה (החלטה, יולי 2026)

התלבטנו בין הגדרת השלבים על ה-`checklist` של משימה בודדת לבין smrtPlan.
**הוכרע: smrtPlan** — כי (א) חלק מהשלבים יעברו לאנשי צוות, ו-smrtPlan
כבר יודע אחראים + תלויות + מסירה אוטומטית; (ב) כבר יש שם תכנון
לפרויקטים אישיים; (ג) המשימות הן ממילא אותה טבלת `tasks`.

### 8.2 מה כבר קיים ומנוצל (אפס/מעט שינוי)

- **משימות = אותה טבלת `tasks`**; קישור דרך `plan_id`, `stage_id`,
  `role_id`, `assigned_to_user_id` (`20260604000100_smrtplan_schema.sql`,
  lifecycle migration).
- **"השלב הנוכחי"** = המשימה ה"מוכנה" הראשונה שלי בתוכנית. חלוקת
  מוכן/חסום/הושלם מחושבת היום ב-`TaskZones.tsx:46-51` (לפי
  `needs.satisfied`), סדר לפי דדליין אפקטיבי `min(due_date, latest_finish)`
  (`TaskZones.tsx:33-41`).
- **מסירה אוטומטית — עובדת מהקופסה.** סיום דרך `POST /tasks/:id/complete`
  מפעיל `releaseDependents` במנוע דרך ה-event bus
  (`on-task-completed.ts` → `engine.ts:585`). **אין מה לשנות שם.**
- **אומדן שעות → משך** כבר קיים: `tasks.estimated_hours`
  (`20260604001200`), המנוע גוזר `duration_days = ceil(estimated_hours /
  שעות-ליום)` (`engine.ts:363-366`).
- **טיוטה שקטה** (`status='draft'`) + `silentPlanIds()` — תוכנית שקטה עד
  אישור.
- **אחראים** — `assigned_to_user_id`, ברירת מחדל = ראשי-התפקיד.

### 8.3 כיוון התזמון — עיקרון-על לשמירה ⚠️

- **מנוע smrtPlan מתזמן אחורה מדדליין** (`end_date`). "מונע-דדליין".
- **בלוק הפוקוס מתזמן קדימה מקצב יומי**. "מונע-קצב".

**דקות-הפוקוס לא מוזנות ל-`duration_days`/למנוע.** "תאריך סיום משוער" הוא
**חישוב-הקרנה קדימה נפרד** = `סכום שעות שנותרו ÷ (דקות-ליום ÷ 60)` → N
ימי-עבודה → `addWorkdays(today, N, blocked)`, על לוח ימי-העסקים המשותף
(§8.3a).
- תוכנית **בלי דדליין** (הרוב האישיים): ההקרנה היא התאריך הראשי (התוכנית
  יכולה לחיות בלי `start_date` — "מאגר").
- תוכנית **עם דדליין**: המנוע ממשיך לסמן סיכון/מסלול-קריטי; ההקרנה מוצגת
  לצידו ("בקצב הזה תגיע/לא תגיע"). לא פותחים מחדש את החלטת המנוע.

### 8.3a מקור ימי-העסקים — util משותף אחד (עיגון, יולי 2026)

**כל** ספירת-ימים בכלי-היום עוברת דרך `src/lib/workdays.ts` — אין לוגיקה
כפולה. הגדרה זהה למנוע (`server/src/modules/smrtplan/engine.ts:112-119`):

- **יום עבודה = שני–שישי ולא בקבוצת החסומים.** שבת **וגם ראשון** חופש
  (`dow === 6 || dow === 0`, `workdays.ts:28-32` / `engine.ts:115-119`).
  הצוות בניו-יורק עובד שני–שישי (ראשון חופש — לא שבוע ישראלי א׳–ה׳).
- **חגים** מטבלת `smrtplan_blocked_days`: שורות גלובליות (`org_id IS NULL`
  = חגי ישראל) + שורות פר-ארגון. שבת/ראשון לא נשמרים (מחושבים).
- קבוצת החסומים מגיעה ללקוח דרך `GET /api/work-calendar`.
- פונקציות רלוונטיות: `isWorkingDay`, `addWorkdays`, `subWorkdays`,
  `workdaysUntil`, `dueUrgency` (+ `DESK_HORIZON_WORKDAYS = 3` לכלל
  3-הימים), `effectiveDeadline`, `sittingWorkdays` (ותק).
- **הקרנת הסיום (§8.3)** קוראת ל-`addWorkdays` מכאן. כלל 3-הימים
  (§3.2/§3.3) קורא ל-`dueUrgency` מכאן. **אפס לוגיקת תאריכים חדשה.**

> החלטה פתוחה: ראשון חופש חל גם על פרויקטים אישיים (ההקרנה מדלגת עליו).
> ברירת המחדל — עקביות עם כל המערכת (שני–שישי). אם יידרש קצב אישי
> שכולל ראשון — יישקל בעתיד, לא בשלב זה.

### 8.4 בונה-תוכנית AI (השינוי בזרימת יצירת התוכנית)

כל פרויקט מתחיל מתוכנית עבודה. שלב היצירה מקבל **מציע-AI**:

- המשתמש מתאר את הפרויקט (טקסט חופשי) + קובע **דקות-ליום** מתוכננות.
- ה-AI מציע: **פירוק לשלבים**, **אומדן שעות לכל שלב** (→ `estimated_hours`),
  ו**חלוקה יומית** לפי קצב ההשקעה.
- מציג **רשימת שלבים** + **תאריך סיום משוער דו-לשוני** (עברית+אנגלית,
  דרך util הפורמט הקיים) מחישוב ההקרנה (§8.3).
- **כל משימה משויכת אוטומטית למקים** (`assigned_to_user_id = creator`);
  מעבר קל על הרשימה לשיוך-מחדש לחבר-צוות (משתמש במנגנון האחראים הקיים).
- התוכנית נולדת **`draft` (שקטה)** — אישור המשתמש (שלבים+שיוכים) מעביר
  ל-`active` ואז היא מופיעה לצוות.

**איפה רץ ה-AI (הוחלט: שניהם):**
1. **AI בתוך האפליקציה** — קריאת Anthropic בשרת (כמו ה-classifier/
   build-brief הקיימים; Sonnet ליצירה). עובר דרך תשתית ה-AI + מעקב-העלות
   הקיים (`daily_ai_budget_usd`, מודלים). זה המסלול לכל משתמש/צוות.
2. **דחיפה מקלוד קוד** — endpoint/ייבוא שמקבל תוכנית מוכנה (שלבים +
   אומדנים + שיוכים) ויוצר אותה כ-`draft`. למשתמש-על שבונה בקלוד קוד.
   שני המסלולים כותבים לאותו מבנה — התוכנית זהה בלי קשר למקור.

- בפרומפט הבנייה: כלל שמירת קישורים עמוקים כלשונם (עקרון מערכת).

**המוח של הבנייה = `project-planning-protocol.md`.** המסמך ההוא (שנבנה
אחרי §8 זה) מגדיר את *התהליך* המלא שהבונה מפעיל: טריאז' גודל, הליכת-תוצר
לאחור + מרשם טכני, שלד הולך, פרה-מורטם, אומדן-מבוסס-פרטים, וסיווג-AI
פר-משימה. הפלט שלו עשיר יותר מ"שלבים + שעות": כל משימה נושאת גם
`definition_of_done`, `ai_tier` (🤖/🤝/👤), `ai_prompt` מוכן, ומשימות-החלטה
עם `affected_by`. שני המסלולים (in-app ו-ייבוא מקלוד קוד) מקבלים את אותה
**סכמת JSON** — המקור הקנוני שלה הוא `project-planning-protocol.md §13`,
והמיפוי לעמודות ב-`smrtplan-focus-integration.md §2/§3`.

### 8.5 התחייבות יומית + מסך הפוקוס

- **`smrtplan_focus` (טבלה חדשה):** `(id, plan_id, user_id, org_id,
  daily_minutes int, active bool, created_at)` + RLS. **פר-אדם-פר-תוכנית**
  — גם איש צוות מתחייב לזמן משלו על אותה תוכנית. **נקבע על מסך התוכנית**
  ב-smrtPlan (שם ההקשר).
- **הצטרפות ליום (הוחלט: אוטומטי):** כל תוכנית עם `active=true` מופיעה
  אוטומטית כל יום כשורת-בלוק ב-/tasks (⏱ NN דק' ▶). מקור:
  `GET /plan/focus-today` (תוכניותיי + השלב הנוכחי + אם בוצע היום).
- **מקביליות (הוחלט: כמה):** כמה תוכניות-פוקוס במקביל, כל אחת בלוק
  נפרד ושורה נפרדת.
- **מסך סשן הפוקוס** — פורק של `MarathonMode`, single-plan:
  - טיימר **יורד** מ-`daily_minutes` (המרתון סופר מעלה).
  - מציג: שם התוכנית + **השלב הנוכחי** (כותרת המשימה המוכנה) + התקדמות.
  - סימון שלב כבוצע תוך כדי → `PATCH /plan-tasks/:id/done` → מסירה.
  - **סיום זמן (הוחלט: מסך חוסם + צליל):** ב-0:00 צליל (Web Audio —
    למרתון אין צליל היום) + שכבת חסימה מלאה עם: `סיימתי להיום` /
    `עוד 5 דקות` (המשך מתועד) / `השלמתי`. בלי התראת דפדפן/רטט בשלב זה.
  - בסוף → כתיבת `focus_sessions`.
- **`focus_sessions` (טבלה חדשה):** `(id, plan_id, user_id, org_id,
  session_date, planned_minutes, actual_minutes, tasks_completed,
  completed_full bool, started_at, ended_at)` + RLS. רצפים, זמן מצטבר
  מול אומדן, היסטוריה.

### 8.6 מה צריך להוסיף בשרת smrtPlan

1. **הרמת לוגיקת מוכן/חסום/סדר ל-util משותף** — היום ב-`TaskZones.tsx`
   (לקוח). לחשוף כך שגם מסך הפוקוס וגם הלקוח יקראו אותו דבר בלי להיסחף.
2. **`GET /plan/:id/focus-stage`** — המשימה המוכנה הראשונה שלי בתוכנית.
3. **`GET /plan/focus-today`** — התוכניות שלי עם התחייבות + שלב נוכחי +
   סטטוס-היום.
4. **`GET /plan/:id/projection`** — חישוב תאריך סיום משוער (הקרנה קדימה).
5. בונה-התוכנית: endpoint ליצירת AI (מסלול 1) + endpoint ייבוא (מסלול 2).

### 8.7 אינטגרציה עם שאר ההגדרות/כלים

- **טוגל במסגרת:** `day_tools.planfocus = { enabled }` (ברירת מחדל
  כבוי — כלי שמפעילים ביודעין).
- **מהיר·3·1:** בלוק-הפוקוס הוא **סקשן נפרד** ב-4-הרשימות (⏱ ▶), לא
  אחד מה-3+1. עובד גם אם 3·1 כבוי (השורה פשוט מופיעה עצמאית).
- **מרתון:** קומפוננטת-בסיס משותפת (טיימר מסך-מלא), אבל נפרד — מרתון =
  ספרינט מהירות, פוקוס = עומק על תוכנית.
- **מסווג size (מהיר/בינוני):** לא רלוונטי — משימות-פלאן לא עוברות
  סיווג-נכנס של הדֶסק ולא מקבלות size (מהיר/בינוני/גדול); מוצגות עם 📋.
  אין התנגשות.
- **סיווג AI של הפרוטוקול (🤖/🤝/👤):** נבדל מ-size של הדֶסק. הוא תכונת-
  משימה (`ai_tier`) שמוצגת כ**תג** על משימת-הפלאן + `ai_prompt` מוכן,
  ומקצרת את ה-`estimated_hours` (זמן אישור, לא זמן יצירה). **לא** הופכת
  את המשימה ל"מהיר" בדֶסק. ("🤖 → מהיר" בפרוטוקול §11 חל על משימות
  שהולכות ישירות ל-smrtTask — טריאז' "משימה"/"פרויקט קטן" §1 — לא על
  משימות בתוך תוכנית מלאה.)
- **תקציב AI / דו-לשוניות / org-scoping / api():** reuse מלא.

### 8.8 שינויי דאטה (כלי הפוקוס)

- `smrtplan_focus` (טבלה חדשה) + RLS
- `focus_sessions` (טבלה חדשה) + RLS
- `day_tools.planfocus` (מפתח ברג'יסטרי — בלי מיגרציה נוספת)
- **אין שינוי ב-`tasks`/במנוע** (הכול כבר קיים: `estimated_hours`,
  אחראים, תלויות, מסירה, טיוטה)

### 8.9 שלבי ביצוע (כלי הפוקוס)

1. **תשתית:** `smrtplan_focus` + `focus_sessions` + טוגל ברג'יסטרי ·
   הרמת לוגיקת המוכן/חסום ל-util משותף.
2. **התחייבות + הקרנה:** שדה דקות-ליום על מסך התוכנית · endpoints
   `focus-stage`/`focus-today`/`projection`.
3. **מסך הפוקוס:** פורק MarathonMode (count-down) · צליל + מסך חוסם ·
   סימון-שלב → מסירה · כתיבת `focus_sessions`.
4. **הצטרפות ליום:** שורת-בלוק ב-/tasks מ-`focus-today` (אוטומטי,
   מקבילי).
5. **בונה-תוכנית AI:** מסלול in-app (Sonnet) + מסלול ייבוא מקלוד קוד ·
   שלבים + `estimated_hours` + חלוקה יומית + תאריך משוער דו-לשוני +
   שיוך-למקים · לידה כ-`draft`.

### 8.10 החלטות שנסגרו (כלי הפוקוס, יולי 2026)

- **מבנה** = תוכנית smrtPlan (לא checklist), שכבת ביצוע מעל (§8.1).
- **בונה-תוכנית** = שניהם — AI in-app + דחיפה מקלוד קוד (§8.4).
- **דקות/יום** = נקבע על מסך התוכנית, פר-אדם (`smrtplan_focus`).
- **הצטרפות ליום** = אוטומטי כל יום.
- **מקביליות** = כמה תוכניות, כל אחת בלוק משלה.
- **סיום זמן** = מסך חוסם + צליל בלבד (בלי התראה/רטט כרגע).
- **השלב הנוכחי** = המשימה המוכנה הראשונה, אוטומטית.
- **כיוון תזמון** = הקרנה קדימה נפרדת; לא נוגעים במנוע (§8.3).
