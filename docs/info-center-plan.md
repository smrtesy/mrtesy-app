# מרכז מידע חכם (smrtInfo) — בדיקה + תכנון סופי

מסמך בדיקה + תכנון. נכתב לאחר סריקה של הקוד, הסכמה של Supabase, ותשתית ה-AI,
ולאחר סגירת ההכרעות המרכזיות עם המשתמש (§7).

המטרה: מרכז מידע אחד שאוסף את כל המידע הרלוונטי מכל מה שהמערכת קולטת — בלי זבל —
ויודע לענות על שאלות בשפה חופשית כמו:

- "מה חברת הביטוח של FPL בארגון מאור?" (עובדה **ארגונית**)
- "מה תאריך התשלום של ביטוח החיים של הבת שלי יהודית?" (עובדה **אישית**)
- "מה הסיסמה החדשה בבנק במשתמש של קופצ'יק?" (סיסמה → **כספת**)

---

## 1. שורה תחתונה (TL;DR)

**רוב התשתית כבר קיימת בפרודקשן.** מה שחסר: שכבת חילוץ עובדות + נקודת-קצה לשאילת
שאלות חופשיות. נבנה כ**אפליקציה חדשה `smrtInfo`**, עם **שני מרכזים — אישי וארגוני**,
**חילוץ היברידי לפי רמת-ביטחון**, ו**חילוץ אוטומטי של סיסמאות מהודעות לתוך smrtVault**.

הערכת מאמץ גסה: **בסיס עובד מקצה-לקצה תוך 3–5 ימי פיתוח**, כי נשענים על רכיבים
שכבר עובדים (pgvector, Voyage embeddings, קליינט Claude, צינור הסיווג, smrtVault).

---

## 2. מה כבר קיים (המצב היום)

### 2.1 אגם המידע הגולמי — `source_messages`
טבלה מרכזית אחת שקולטת את **כל** הערוצים: `gmail`, `gmail_sent`, `google_drive`,
`google_calendar`, `whatsapp`, `whatsapp_echo`, `sms`. שומרת את הטקסט המלא ב-
`raw_content` (~3000 תווים) ו-`body_text` (~1000 תווים), וגם `subject`, `sender`,
`source_url`, ו-`metadata` (jsonb). זה בדיוק "כל המידע שהאפליקציה מקבלת ועוברת עליו".
בנוסף: `whatsapp_messages.body_text` (כולל תמלול אודיו ב-Gemini ו-OCR לתמונות) ו-
`sms_messages.body_text`.

### 2.2 תשתית חיפוש סמנטי (RAG) — כבר בפרודקשן
- **pgvector 0.8.0 מותקן** (HNSW / ivfflat).
- **Voyage embeddings**: `server/src/services/voyage.ts` — `embedText()` → וקטור
  1024-מימדים (`voyage-4`), עם רישום עלות ל-`ai_usage`.
- `knowledge_base` + `match_knowledge_base(_org)` — חיפוש קוסינוס
  (`supabase/migrations/20260527030000_knowledge_base.sql`).
- **מגבלה:** משמש היום רק ל"כתוב-פעם-ואז-שכפל" (ניסוח תשובות חוזרות). **אין** שאילת
  שאלות חופשית על הקורפוס.

### 2.3 עוזרי AI קיימים
- **Anthropic** (`server/src/anthropic.ts`): `simpleCall`, `cachedCall`,
  `parseJsonResponse` (haiku/sonnet/opus), עם רישום עלות ל-`ai_usage`.
- **Gemini** (`server/src/gemini.ts`): תמלול אודיו, OCR, וחילוץ טקסט מ-PDF
  (`extractDocumentText`) — קריטי לקריאת פוליסות/חשבונות שמגיעים כקבצים.
- **צינור הסיווג** (`ai-process` edge + `part3`/router): כבר מסווג כל הודעה
  ACTIONABLE / INFORMATIONAL / SPAM — נקודת-תפיסה טבעית לחילוץ.

### 2.4 חילוץ עובדות חלקי שכבר קיים
- `project_briefs.pending_facts` / `verified_facts` (jsonb) — עובדות AI עם שער-אישור.
- `project_information_items` + `projects.info_summary` — **"Info Center" קיים אבל
  ברמת פרויקט בודד** (`20260610190100_info_center.sql`), לא גלובלי.

### 2.5 smrtVault — כספת הסיסמאות
`smrtvault_credentials` (מטא-דאטה בגלוי) + סיסמה מוצפנת ב-Supabase Vault. חשיפה רק
דרך `/vault/credentials/:id/reveal`, נרשמת ל-`smrtvault_access_log`. פותר את
"קופצ'יק בבנק".

### 2.6 זמין בלי הקצאה נוספת
`pgvector`, `pg_cron`, `pg_net`, `supabase_vault` מותקנים; `pgroonga` (טקסט מלא בכל
השפות כולל עברית) ו-`pg_trgm` זמינים להתקנה.

## 3. הפער — מה חסר

| מה שביקשת | מה חסר |
|---|---|
| מרכז אחד שאוסף הכל, בלי זבל | מאגר עובדות מובנה ומנוקה |
| לשאול שאלה חופשית ולקבל תשובה | נקודת-קצה "שאל את המידע שלי" (retrieval + ניסוח) |
| תשובה מדויקת עם מקור | הצמדת ציטוט + deep link לכל תשובה |
| הבחנה בין אישי לארגוני | מנגנון סיווג scope (§5.2) |
| גם סיסמאות (קופצ'יק) | חילוץ אוטומטי לכספת + גשר מאובטח (§5.3) |

## 4. הכרעות שנסגרו

1. **אריזה:** אפליקציה חדשה **`smrtInfo`** (מסך נפרד לצד הכספת).
2. **טווח:** **שני מרכזים — אישי + ארגוני.** אישי: כל משתמש רואה את שלו. ארגוני:
   מנהלי הארגון. כרגע המשתמש היחיד הוא מנהל מאור → רואה הכל במקום אחד, אבל מודל
   הנתונים מפוצל מהיום הראשון.
3. **חילוץ:** **היברידי לפי רמת-ביטחון** (ביטחון גבוה → חי; נמוך → "לא מאומת").
4. **סיסמאות:** סיסמה שמוזכרת בהודעה **מחולצת אוטומטית ל-smrtVault** (לא ל-`info_facts`).

---

## 5. ארכיטקטורה — שכבה דקה מעל הקיים

### 5.1 שכבה A — חילוץ עובדות (בזמן קליטה)
מתחברים לצינור הסיווג. כל פריט INFORMATIONAL (או ACTIONABLE עם עובדה) עובר קריאת
חילוץ ל-Claude (`simpleCall` + `parseJsonResponse`) שמחזירה:

```
{ entity, attribute, value, effective_date?, confidence (0–1),
  scope: "personal" | "org" | "unclassified",
  is_secret: boolean,          // אם זו סיסמה/קוד → נתיב הכספת, לא info_facts
  source_message_id, source_url (deep link מלא), language }
```

- **PDF/תמונות:** קודם `extractDocumentText` / `performImageOcr` (Gemini) → טקסט → חילוץ.
- **חילוץ היברידי:** `confidence ≥ סף` → `verified=true` (חי וסמכותי). אחרת →
  `verified=false` ("לא מאומת"), מוצג עם אזהרה ומוצע לאישור בלחיצה.
- **מנקה זבל:** רק INFORMATIONAL + סף ביטחון; מדלגים SPAM/NOISE, שיווק, ו-OTP
  (כבר לא נשלח ל-AI).
- **דדופ + גרסאות:** מפתח (entity, attribute, scope) — החדש גובר, הישן ל-
  `info_fact_history`. כך "הסיסמה **החדשה**" מחליפה את הישנה עם שמירת מתי השתנתה.

### 5.2 סיווג אישי / ארגוני — "פרופיל הקשר"
המנגנון שמפצל תיבה אחת לשני מרכזים:
1. **פרופיל הקשר** (`info_context_profile`, עריך במסך הגדרות): *הארגונים שלי*
   (מאור, דומיין `maor.org`, ספקים: FPL…), *המשפחה שלי* (יהודית=בת…), *החשבונות
   שלי* (אילו כתובות אישיות מול ארגוניות).
2. **תיוג AI** — החילוץ מציע `scope` על סמך הפרופיל + החשבון שאליו הגיעה ההודעה
   (`maor.org` מול `kinus.info`).
3. **נפילה ל"לא מסווג"** — מה שלא בטוח מופיע בשני המרכזים עם כפתור "שייך לאישי/ארגוני";
   השיוך מלמד את הפרופיל.

### 5.3 סיסמאות → smrtVault (אוטומטי)
כשהחילוץ מזהה `is_secret` (למשל "הסיסמה החדשה בבנק היא…"):
- נכתב **ישר ל-smrtVault** (create/rotate של `smrtvault_credentials`, הסיסמה ל-Vault).
- **לעולם לא נכנס ל-`info_facts` ולא נרשם בטקסט גלוי.**
- ב-`info_facts` נשמרת רק הצבעה לא-רגישה ("קיימת סיסמה לבנק/קופצ'יק בכספת").
- שאלה על סיסמה → כרטיס עם כפתור "חשוף" דרך ה-reveal הקיים (נרשם ל-
  `smrtvault_access_log`).

### 5.4 שכבה B — אחזור
`match_info_facts(query_embedding, scope, p_user_id/p_org_id, threshold, count)` על
משקל `match_knowledge_base`. **חיפוש היברידי:** וקטורי (משמעות) + מילולי עברית-מודע
(`pg_trgm`/`pgroonga`, או `ILIKE` fallback) לתפיסת שמות מדויקים ("FPL", "יהודית",
"קופצ'יק"). למנהל — חיפוש חוצה אישי+ארגוני.

### 5.5 שכבה C — שאלה + תשובה (UI)
- `POST /api/info/ask`: embed לשאלה → אחזור top-K → `simpleCall` עם הוראות קשיחות:
  **לענות רק מהמקורות, לצטט מקור עם deep link מילה-במילה, ולומר "לא נמצא" אם אין.**
- מסך `smrtInfo`: תיבת חיפוש אחת (מכווצת כברירת מחדל, לפי עקרון ה-UI המינימלי) →
  כרטיס תשובה עם ציטוטים וקישורים; שתי לשוניות "אישי"/"ארגוני"; רשימת עובדות
  לעיון/עריכה/מחיקה + מסך סקירה קטן ל"לא מאומת".
- רישום pane ב-`src/lib/panes/registry.tsx`.

### דוגמאות מקצה לקצה
1. "חברת הביטוח של FPL במאור?" → מייל מחברת ביטוח → עובדה
   `{entity:"FPL", attribute:"insurer", scope:"org"}` → תשובה + קישור למייל.
2. "מתי תשלום ביטוח החיים של יהודית?" → פוליסה (PDF) → OCR → עובדה
   `{entity:"ביטוח חיים – יהודית", attribute:"payment_date", scope:"personal"}` → תשובה + קישור.
3. "הסיסמה החדשה בבנק של קופצ'יק?" → אם הוזכרה בהודעה נחלצה אוטומטית לכספת → כרטיס
   עם כפתור "חשוף" מאובטח.

## 6. מה זה דורש (רשימת עבודה)

1. **מיגרציה**: `info_facts` (עמודות מעל + `embedding vector(1024)` + HNSW + RLS) +
   `info_fact_history` + `info_context_profile` + `match_info_facts`.
   RLS: עובדות `personal` → לפי `user_id`; `org` → לחברי הארגון עם תפקיד ניהול.
   (אופציונלי: הפעלת `pgroonga`/`pg_trgm`.)
2. **רישום אפליקציה**: הוספת `smrtinfo` ל-`apps` + `app_status`, כמו
   `20260713100000_register_smrtvault.sql`.
3. **מודול שרת** `server/src/modules/smrtinfo`: `/info/ask`, `/info/facts` (CRUD),
   `/info/context-profile`, וה-hook לחילוץ. שימוש חוזר ב-`embedText`, `simpleCall`,
   `parseJsonResponse`, `ai_usage`, ובנתיב create/rotate של smrtVault.
4. **חיווט לצינור הסיווג**: הוספת קריאת חילוץ ב-`ai-process` (edge) ו/או במסלול
   ה-Express. ⚠️ מודעות דו-מנועית: edge = deploy ידני.
5. **פרונטאנד**: מסך smrtInfo + pane + מסך פרופיל הקשר + מפתחות i18n (he/en יחד).
6. **Backfill חד-פעמי**: חילוץ+embedding על היסטוריית `source_messages`.
7. **סביבה**: `VOYAGE_API_KEY` כבר מוגדר (knowledge_base משתמש בו) — לוודא שקיים
   בשני המנועים אם החילוץ רץ ב-edge.

---

## 7. אבטחה ופרטיות
- `info_facts` / `info_context_profile` עם RLS: אישי לפי `user_id`, ארגוני לחברי
  הארגון עם תפקיד ניהול. גישת כתיבה רק דרך שרת (service-role).
- **סיסמאות לעולם לא נשמרות ב-`info_facts` ולא נשלחות ל-LLM מעבר לרגע הזיהוי;**
  ערך הסיסמה נכתב ישר ל-Vault. חשיפה רק דרך ה-reveal הקיים, מתועדת.
- OTP/קודים חד-פעמיים כבר לא נקלטים ל-AI.
