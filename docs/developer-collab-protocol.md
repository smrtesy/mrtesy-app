# פרוטוקול שילוב מתכנת (איימן) — שיטת עבודה

מסמך שיטה, יולי 2026. **נלווה ל-`project-planning-protocol.md`.** בעוד
פרוטוקול התכנון עונה על "איך מתכננים פרויקט", המסמך הזה עונה על
"**איך משלבים מתכנת (איימן) בפרויקט מתוכנן** — בלי לאבד את המהירות של
לעבוד לבד עם Claude Code, ובלי לצבור חוב ארכיטקטוני". רב-פרויקטלי:
חל על כל פרויקט עתידי שבו איימן משתתף.

מסמכים צמודים: `project-planning-protocol.md` (בניית התוכנית),
`PROJECT_GUIDE.md` (ארכיטקטורת המערכת לבעלים+מתכנת), `CLAUDE.md`
(ספר ההפעלה + פרוטוקול ה-pre-push), `workclock-plan.md` (שכבת הזמן).

## 1. התוצר

מסילת עבודה מוסכמת שמאפשרת לאיימן לתרום קוד מהיום, שבה:
- **הידע עובר דרך הכרטיס והריפו — לא דרך חנוך בעל-פה.**
- **הקונבנציות נאכפות אוטומטית** (Claude Code קורא `CLAUDE.md`), לא
  נשענות על זיכרון.
- **הבקרה מבוססת-תוצאה**: PR שעבר שער + אומדן-מול-בפועל פרטי למנהל.
  **בלי צילומי מסך, בלי ניטור נוכחות.**

## 3. המודל הניהולי (החלטת המשתמש, יולי 2026)

1. **Backlog מסודר, לא הקצאה יומית.** איימן מקבל **רשימת משימות לפי
   סדר עדיפות** ומושך את הבאה בתור. לא כותבים לו "היום תעשה X". הוא
   מנהל את הזמן שלו; אנחנו מנהלים את **הסדר**.
2. **האומדן פרטי למנהל.** לכל משימה חנוך רושם אומדן שעות — **איימן
   לא רואה אותו.** הסיבה: אומדן גלוי הופך ליעד שמעגנים אליו או
   "משחקים" מולו; אנחנו רוצים סיגנל ניהול נקי, לא לחץ.
3. **מדידת פריון = אומדן מול בפועל.** הזמן-בפועל נלכד בשעון-העצמי
   (`workclock` → `work_task_spans`). המנהל רואה, פר-משימה, כמה
   שוער מול כמה לקח. זה הסיגנל היחיד — לא נוכחות, לא צילומים.
4. **המיזוג הוא של חנוך — לפי סיכום בעברית, לא לפי קריאת קוד.**
   חנוך *לא קורא קוד* (בשונה מאיימן). לכן איכות הקוד היא באחריות
   איימן ונאכפת אצלו (pre-push protocol + CI); ולכל PR נוצר **סיכום
   בעברית** ברמת-מוצר (מה השתנה, למה, וסיכון אם יש). חנוך קורא את
   הסיכום — "אם זה נראה לי, ממזג". "בוצע" = PR מוזג ל-`main` עם CI
   ירוק שעונה על הגדרת-הבוצע.
5. **תקציב הניהול של חנוך: עד 15 דקות ביום — קשיח.** כל השיטה
   מתוכננת סביב האילוץ הזה. המנוף המרכזי לעמוד בו הוא **בהירות
   התצוגה**: ככל שהמידע שמוצג למנהל מסודר וממוקד יותר — כך 15 הדקות
   מספיקות. לכן יש **דוח מנהל יומי מאוחד** (ק8) שמציף *רק* את מה
   שדורש את חנוך, וכתיבת הכרטיסים מוזלת ל-Claude (brief של 2 שורות)
   כדי שלא תאכל את התקציב.

## 4. מסלול חיי-משימה (מקצה לקצה)

```
תכנון פרויקט (חנוך + Claude, לפי project-planning-protocol)
      ↓  מפיק משימות מסודרות ב-smrtPlan, כל אחת עם אומדן פרטי
Backlog מסודר לפי עדיפות  →  איימן מושך את הבאה בתור
      ↓
איימן עובד ב-Claude Code (CLAUDE.md נאכף)  +  שעון-עצמי סופר בפועל
      ↓
PR בגיטהאב  →  שער הקוד (של איימן): build + pre-push protocol + CI
      ↓  Claude מפיק סיכום עברית ל-PR (רמת-מוצר)
חנוך קורא את הסיכום בתוך "נכנס/משימות"  →  אם תקין, ממזג
      ↓  (CODEOWNERS מבטיח שליבה מחייבת את מיזוג חנוך)
מיזוג ל-main  →  smrtPlan סוגר את המשימה
      ↓
המסך היחיד: אומדן מול בפועל (פרטי) + לוג-AI "מה נעשה" + PR למיזוג
```

**מה איימן רואה:** כרטיס מלא (הקשר, צעדים, חומרים עם קישורים מלאים,
הגדרת-בוצע), והמקום שלו בסדר. **מה הוא לא רואה:** האומדן.
**מה חנוך רואה — הכול במסך אחד (נכנס/משימות):** PR-ים למיזוג עם סיכום
עברית, אומדן מול בפועל, ולוג "מה נעשה".

### 4א. דוח המנהל היומי — משתלב בנכנס/משימות (ק8)

**החלטת המשתמש: מסך אחד שבו חנוך מנהל את כל היום שלו.** לכן הדוח *אינו*
מסך נפרד ואינו דייג'סט נדחף — הוא **כלי-יום** (day-tool) חדש שנתלה על
נקודות-העגינה של שולחן המשימות הקיים (`docs/day-tools-plan.md §2.4`),
לפי הכללים: toggle אחד ב-`user_settings.day_tools`, רשומה ב-
`src/lib/smrttask/day-tools.ts`, ועיצוב קומפקטי (כפתור/סקשן שקט, לא
סרגל קבוע — `CLAUDE.md`, עקרון ה-UI הקומפקטי). פריטי איימן זורמים
**לתוך** השולחן של חנוך, כך שהם חלק מהיום שלו ולא מסך שצריך לזכור
להיכנס אליו.

עיקרון: מציף **רק את מה שדורש את חנוך**, מסודר לפי דחיפות. ארבעה
בלוקים על השולחן:

1. **ממתין להחלטתך (מיזוג)** — PR-ים שממתינים למיזוג, כל אחד עם
   **סיכום העברית** שלו + קישור ל-PR. מיזוג = פעולה מהשולחן. ליבה
   מחייבת את חנוך (CODEOWNERS); מבודד — לפי הגדרה, אפשר שאיימן ימזג לבד.
2. **מה איימן עשה** — לוג-AI פר-משימה שנסגרה: 2–3 שורות + קישור ל-PR.
3. **חריגות אומדן** — רק משימות שחרגו מהותית מהאומדן (מעל סף מוגדר),
   לא כל שורה. סיגנל ל"תקוע" או "האומדן שלי לא מדויק".
4. **הבא בתור** — משימות שממתינות לאומדן/סידור *ממך*, כדי שאיימן לא
   ייחסם. זה החלק שאם תדלג עליו — איימן נעצר.

הכול על שולחן אחד, סָריק ב-15 דקות. אם משהו לא דורש פעולה — הוא לא
מופיע.

## 5. חבילת אונבורדינג (משימה ראשונה = שלד הולך)

מטרה: איימן פרודוקטיבי תוך יום-יומיים, בלי לשרוף את הזמן של חנוך.

1. **גישות** (צ'קליסט): הרשאת GitHub לריפו · Claude Code · משתני
   הסביבה שהוא צריך. **מה הוא לא מקבל:** סודות production, מפתחות
   API בתשלום, גישת כתיבה ל-`main` (עובד רק ב-PR).
2. **מסלול קריאה מודרך:** `PROJECT_GUIDE.md` (מה המערכת ושלושת
   המנועים) → `CLAUDE.md` (הקונבנציות + pre-push) → מסמך זה.
3. **משימה ראשונה = פיצ'ר מבודד קטן מקצה-לקצה** (לא ליבה): משימה →
   קוד ב-Claude Code → pre-push → PR → מיזוג. לומד את כל הצינור על
   משהו נמוך-סיכון. זה השלד ההולך של השילוב עצמו.
4. **קובץ שפת-מפעיל** (נספח א'): איימן שם שורה אחת → Claude Code עונה
   וכותב לו כרטיסים באנגלית.

## 10. קצב ותאריך

- **תקציב ניהול של חנוך: עד 15 דקות/יום — קשיח (§3.5).** מתוקצב כך:
  review ל-PR של ליבה + סריקת דוח המנהל היומי (ק8) + סידור/אומדן של
  הבא-בתור. כתיבת הכרטיסים עצמה מוזלת ל-Claude כדי לא לחרוג. אם 15
  הדקות לא מספיקות בפועל — הסימן הוא ש**התצוגה לא מספיק ברורה** או
  ש-CODEOWNERS מפנה יותר מדי לחנוך; מתקנים שם, לא מוסיפים זמן.
- **קיבולת איימן: 30 שעות/שבוע לכלל הפרויקטים.** הפריון פר-פרויקט
  ניתן לאמידה רק אחרי שמחליטים מה הפרויקטים ומה חלקו של איימן בכל
  אחד — ואז חולקים את ה-30 שעות ביניהם.
- **תאריך פר-פרויקט** נגזר בתחנה 11 של פרוטוקול התכנון:
  `סכום אומדנים + באפר 20% ÷ (קיבולת שהוקצתה לפרויקט)`, מוצג
  דו-לשונית (עברית + אנגלית). מסמך זה מגדיר את **המנגנון**; כל פרויקט
  מזין אליו scope.
