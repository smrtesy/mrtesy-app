# smrtPlan — סימולציית תוכנית ("מה העובד יראה כל יום") + מעבר סקירה

מסמך תכנון, יולי 2026. הצעה לפיצ'ר ב-smrtPlan שמאפשר **לדפדף בתוכנית יום-יום
קדימה** ולראות בדיוק מה כל עובד יקבל במשימות שלו בכל יום — לפני שהתוכנית רצה —
וכן **מעבר סקירה מסומן** לתיקון מרוכז. נכתב אחרי בניית פרויקט-הווידאו הראשון,
שבו רוב חיכוכי-התכנון נבעו מחוסר יכולת לראות מראש את חוויית-העובד.

מסמכים צמודים: `project-planning-protocol.md` (מתודולוגיית התכנון),
`smrtplan-focus-integration.md` (מנוע הפוקוס והתזמון).

---

## 0. תקציר מנהלים

מוסיפים ל-smrtPlan **מצב סימולציה**: המשתמש בוחר יום ומדפדף קדימה/אחורה, ורואה
לכל אדם את **תור המשימות שהיה מקבל באותו יום** — בדיוק כפי שיראה במסך ה"נכנס"
שלו. זה הופך אימות-תוכנית מניחוש ("מה לוי יראה בשלישי?") להליכה חזותית.

הפיצ'ר נשען על חלק אחד שחסר היום: **מתזמן-קיבולת קדימה** — "אדם אחד, X שעות
ביום, משימה-משימה, לפי תלויות וחסימות חגים". המנוע הקיים הוא CPM (תלויות +
דדליין אחורה); הוא לא מסדר עבודה לזרם יומי פר-אדם. הסימולציה = להריץ מתזמן כזה
קדימה ולהציג את התוצאה יום-יום.

בונוס משלים: **מעבר סקירה** — לסמן פר-משימה "מה לשנות", לצבור לרשימה ולייצא
לתיקון מרוכז (מיושם היום כטבלת CSV חיצונית; אפשר להטמיע בהמשך).

---

## 1. הבעיה (עדות מהשטח)

בבניית פרויקט-הווידאו הראשון, שאלות שחזרו שוב ושוב:
- "למה התאריך של המשימה הראשונה הוא היום ולא לפני שלישי?"
- "מה בדיוק לוי יראה ביום שהוא מתחיל?"
- "אם המשימה לוקחת 6 שעות והוא עובד 3 ביום — כמה ימים זה?"

כל אלה נענו רק אחרי **חישוב ידני** של לוח קדימה (סימולציה שנעשתה מחוץ למערכת).
המערכת לא ידעה להראות את זה, כי:

1. **תצוגת הלוח מציגה "דדליין אחרון"** (תזמון-אחורה מהיעד), לא "מתי מתחילים".
2. **המנוע לא מסדר עבודה לזרם יומי פר-אדם** — הוא CPM, לא resource-leveling.
   משימה = יום קלנדרי לפי תלויות, בלי לסדר את המשימות של אותו אדם בזו-אחר-זו
   לפי הקיבולת היומית שלו.

התוצאה: אי אפשר לענות על השאלה הבסיסית ביותר — *"תראה לי את יום שלישי של לוי"*.

---

## 2. המטרה

מצב סימולציה בעמוד התוכנית:
- בורר יום (או "יום 1 / יום 2 …" יחסית להתחלה) עם **חיצי דפדוף קדימה/אחורה**.
- לכל אדם בתוכנית: **התור שלו באותו יום** — המשימה/ות שהיו "מוכנות" ופתוחות לו,
  בדיוק בפורמט של מסך העבודה שלו (כותרת, כמה מהיום היא לוקחת, מה חוסם).
- סימון ימים חסומים (חגים, ראשון, ימי-אי-עבודה של האדם).
- **בלי לשנות נתונים** — הסימולציה היא הקרנה, לא כתיבה.

מבחן ההצלחה: מתכנן פותח את הסימולציה, מדפדף מהיום הראשון עד האחרון, ורואה את
כל מסע-העבודה של כל אדם — ותופס בעיות ("ביום הזה לוי חסום ומחכה", "פה יש קפיצה
של שבוע בגלל החג") **לפני** ההפעלה.

## 3. הליבה — מתזמן-קיבולת קדימה

זה החלק היחיד שדורש עבודת-מנוע אמיתית. הקלט לכל אדם: קיבולת יומית
(`smrtplan_capacity.hours_per_day`), ימי-עבודה (`work_days`), וחסימות חגים
(`src/lib/workdays.ts`). האלגוריתם (זהה לזה שחושב ידנית בפרויקט-הווידאו):

```
לכל משימה, בסדר טופולוגי (לפי תלויות):
  duration_days = ceil(estimated_hours / hours_per_day של האחראי)
  earliest = max( הזמן-הפנוי-הבא של האחראי,
                  יום-אחרי-סיום של כל תלות,
                  אילוץ-התחלה-מוקדמת אם יש )
  start  = יום-העבודה הבא של האחראי מ-earliest (מדלג חגים/ימי-אי-עבודה)
  finish = start + (duration_days-1) ימי-עבודה של האחראי
  הזמן-הפנוי-הבא של האחראי = יום-אחרי-finish
```

עקרונות:
- **פר-אדם, משימה אחת בכל רגע** (resource-leveling אמיתי) — זה מה שהמנוע
  הנוכחי לא עושה.
- מכבד **ימי-עבודה שונים לכל אדם** (לוי שני–חמישי, חנוך שני–שישי) וחגי ישראל.
- **המתנות חיצוניות** (`external_wait_days`, תחנה 8 בפרוטוקול) מוצגות כזמן-רקע
  שלא שורף סשן — יום קלנדרי אבל לא יום-אדם.

**חשוב:** זה **מתזמן חדש נפרד**, לא שינוי במנוע ה-CPM הקיים. שני המנועים חיים
זה לצד זה — CPM נותן "דדליין אחרון", המתזמן-קדימה נותן "מסלול ריאלי מהיום".
(עולה בקנה אחד עם "לא נוגעים במנוע התזמון" ב-`smrtplan-focus-integration.md §0`.)

## 4. הצד הטכני

### 4.1 API (חדש, קריאה בלבד)
```
GET /plans/:id/simulation
  → {
      start_date, end_date,
      people: [{ user_id, name, hours_per_day, work_days }],
      days: [
        { date, blocked: bool, blocked_reason?,      // חג / ראשון / אי-עבודה
          per_person: [
            { user_id,
              active:   [{ task_id, title, hours, day_of_span, span_days }],  // עובד עליה היום
              waiting:  [{ task_id, title, blocked_by }],                     // מוכנה אך חסומה
              idle:     bool }                                                // אין עבודה זמינה
          ] }
      ]
    }
```
כל החישוב בצד השרת, מהנתונים הקיימים (`tasks`, `smrtplan_dependencies`,
`smrtplan_capacity`, לוח החגים). אין טבלה חדשה, אין כתיבה.

### 4.2 קוד
- מודול חדש `server/src/modules/smrtplan/simulate.ts` — המתזמן-קדימה (פונקציה
  טהורה: קלט tasks+deps+capacity+holidays → לוח יומי). ניתן לבדיקת-יחידה.
- שימוש חוזר ב-`workdays.ts` לחסימות.
- ה-endpoint עוטף את המודול.

### 4.3 UI
- טאב רביעי בעמוד התוכנית ("סימולציה"), לצד רשימה / ציר / יומן.
- בורר-יום עם חיצי ‹ ›, "יום N מתוך M", ותאריך דו-לשוני.
- טור לכל אדם; כרטיס משימה בפורמט זהה למסך-העבודה (density של `PlanEffortDetail`).
- ימים חסומים מסומנים ואפשר לדלג עליהם.
- ברירת מחדל: היום הראשון של התוכנית.

## 5. שילוב "מעבר הסקירה" (החלק המשלים)

בזמן שמדפדפים בסימולציה ותופסים בעיה — רוצים לסמן אותה. שתי דרגות:

- **דרגה 0 (קיים היום):** טבלת סקירה חיצונית (CSV) — שורה לכל משימה, עמודה
  "מה לשנות". המשתמש ממלא ומייצא; קלוד מחיל מרוכז.
- **דרגה 1 (הטמעה עתידית):** "הערת-שינוי" פר-משימה בתוך האפליקציה — כפתור קטן
  על המשימה פותח שדה, ההערות נצברות לרשימה שניתן לייצא כ-payload מובנה שקלוד
  מחיל. זה הופך את לולאת-התיקון לחלק מהמערכת.

הדרגה-1 קטנה יחסית (שדה + טבלת `plan_review_notes` + ייצוא), ואפשר להוסיף אחרי
הסימולציה.

## 6. פירוק לשלבים (מדורג, כל שלב בר-שילוח)

1. **מתזמן-קדימה + endpoint** (`simulate.ts` + `GET /plans/:id/simulation`).
   בדיקת-יחידה מול הלוח שחושב ידנית בפרויקט-הווידאו (אמת-מידה מוכנה).
2. **מסך הסימולציה** — טאב + בורר-יום + טורים-לפי-אדם.
3. **ליטוש** — ימים חסומים, המתנות חיצוניות, תצוגת "חסום/ממתין/פנוי".
4. **(רשות) מעבר-סקירה בתוך האפליקציה** — דרגה 1.

שלב 1 הוא הלב וגם הכי שווה לבדיקה — הוא גם מה שיאפשר בעתיד להציג תאריכים
ריאליים על הלוח עצמו (במקום "דדליין אחרון").

## 7. סיכונים ושאלות פתוחות

- **חפיפה עם מנוע-הפוקוס בפיתוח:** הצוות כבר בונה `smrtplan_focus` / `focus_sessions`
  / `planned_for`. צריך לוודא שהמתזמן-קדימה **מזין** את אותו רעיון ולא מכפיל
  אותו — אולי `planned_for` הוא בדיוק הפלט שהסימולציה מציגה. **לתאם לפני בנייה.**
- **מקור-אמת אחד לתאריכים:** אם המתזמן-קדימה נכון, האם הוא צריך גם *לכתוב*
  תאריכי-יעד (במקום החישוב הידני של היום)? החלטה: שלב 1 קריאה-בלבד; כתיבה זו
  החלטה נפרדת.
- **ריבוי אנשים ומשימות מקבילות:** המתזמן צריך לטפל נכון בשני אנשים שעובדים
  במקביל (חנוך+לוי) ובמשימה שממתינה לפלט של האחר.
- **דיוק מול פשטות:** לא לנסות לדמות הכל (מחלות, חופשות אד-הוק) — סימולציה
  טובה-מספיק שמבוססת על קיבולת+תלויות+חגים עדיפה על מודל מושלם שלא נבנה.

## 8. המלצה

לבנות בסדר של §6, אחרי **תיאום קצר עם work-stream של מנוע-הפוקוס** (סיכון §7
הראשון). שלב 1 (המתזמן) הוא ההשקעה המשמעותית ובעל הערך הגבוה ביותר — הוא גם
פותר את שורש הבלבול על התאריכים, לא רק את הסימולציה. עד אז, מעבר-הסקירה (§5
דרגה 0) כבר עובד כטבלת CSV.
