# תכנון: Stop hook מאוחד עם ניתוב לפי משתמש (smrtTask + smrtPlan)

מסמך תכנון לאישור. מתאר מנגנון אחד, זהה, שיושב **בשני הרפוים**
(`mrtesy-app` ו-`video-lab`) ובורר בין שתי זרימות דיווח לפי **מי פתח את
הצ'אט** — חשבון Claude Code או זהות שהוגדרה ידנית בתחילת הסשן.

> **סטטוס:** טיוטה לאישור. סעיף [שאלות פתוחות](#שאלות-פתוחות) מסמן מה עוד
> צריך אישור לפני בנייה. אין כתיבה ל-DB או לפרודקשן עד אישור.

---

## המטרה

היום קיימים שני מנגנונים נפרדים:

- **`mrtesy-app`** → הצעת סשן ל-**smrtTask**: כל שיחה יוצרת/מרעננת "הצעה"
  בתיבה, עם נושא, סיכום, צעד המשך, קישור עמוק, ו(מאז השינוי האחרון) **זהות
  חשבון Claude**. Endpoint: `POST /api/claude-session/proposal`.
- **`video-lab`** → דיווח סטטוס ל-**smrtPlan**: מתחבר למשימה שמסומנת
  `in_progress` אצל המשתמש, ומעדכן את מנהל התוכנית. Endpoint:
  `POST /api/claude-session/task-report`.

**הרצוי:** מנגנון אחד בשני הרפוים, שמחליט לפי המשתמש איזו זרימה להפעיל.

---

## עקרון הזיהוי

הסשן יודע תמיד את אימייל הכניסה של חשבון Claude Code דרך משתנה הסביבה
`CLAUDE_CODE_USER_EMAIL` (זמין לכל hook, בלי קריאת רשת). על בסיסו:

### מקרה א' — חשבון המנהל (`CLAUDE_CODE_USER_EMAIL == chanoch@maor.org`)
- **לא נשאלת** שאלת "מי אתה?" — ידוע שזה המנהל.
- חשבון הפלטפורמה לפתרון: **תמיד `chanoch770@gmail.com`** (mapping קבוע).
- זרימה: **שילוב** (ראה למטה).

### מקרה ב' — כל חשבון Claude אחר (חשבונות משותפים לעובדים)
מכיוון שחשבון Claude משותף למספר עובדים, אימייל הכניסה **לא** מזהה את
האדם. לכן בתחילת הצ'אט הסוכן שואל **"מה האימייל שלך ב-smrtTask?"** ומציג
רשימה שמורה:

```
מה האימייל שלך ב-smrtTask?
1. chanoch@maor.org
2. lw@maor.org
... (כל מי שברשימה)
או: הקלד את אימייל ה-smrtTask שלך — יישמר לרשימה לפעם הבאה.
```

**שפת השאלה:** נשאלת ב**עברית או אנגלית לפי השפה שבה כתב המשתמש** בהודעתו
(אותו עיקרון של i18n בשאר המערכת). העובד מקליד את **אימייל ה-smrtTask שלו**
— זהו האימייל שנפתר ישירות מול הפלטפורמה (ראה שאלה 2, מאושר).

- בחירת **המנהל (`chanoch@maor.org`)** → זרימת **שילוב** (זהה למקרה א',
  פתרון מול `chanoch770@gmail.com`).
- בחירת **עובד אחר** → זרימת **video-lab** (סיכום למשימת העובד + הצעה
  למנהל).

## שתי הזרימות

### זרימת שילוב (מנהל)
> "אם יש משימה פתוחה — עדכן אותה; אם לא — הגש הצעה, כמו שעובד היום."

מנוצל שכבר קיים: ה-endpoint של `task-report` מחזיר `attached:false` כשאין
משימה `in_progress`. לכן ה-hook:

1. שולח ל-`/api/claude-session/task-report`.
2. אם התשובה `attached:true` → סיימנו (המשימה עודכנה).
3. אם `attached:false` → נופל חזרה ל-`/api/claude-session/proposal`
   (הצעה, בדיוק כמו היום, כולל זהות חשבון Claude).

**אין צורך בלוגיקה חדשה בשרת לזרימה הזו** — רק תזמור בצד ה-hook.

### זרימת עובד (video-lab)
> "סיכום במשימה לעובד, ודיווח כהצעה למנהל."

1. שולח ל-`/api/claude-session/task-report` עם זהות **העובד** → מתחבר
   למשימת ה-`in_progress` של העובד (סיכום במשימה שלו).
2. **בנוסף** מגיש **הצעה** בתיבת ה-smrtTask של **המנהל**, שמסכמת את דיווח
   העובד + קישור לסשן.

> ⚠️ **שינוי מהקיים:** ה-video-lab היום **מתריע** (notification) למנהל, לא
> מגיש הצעה. הבחירה שלך: **הצעה** בתיבת המנהל (פריט פעולה). זה דורש תוספת
> קטנה בשרת — ראה [שינויי שרת](#שינויי-שרת).

## ארכיטקטורה

### 1. hook מאוחד בשני הרפוים (repo דינמי)
- מאחדים את `smrttask-session-proposal.sh` + `post-session-summary.sh`
  (+ `build-session-proposal.mjs`) עם לוגיקת ה-video-lab לכדי סט אחד.
- **שם הרפו נגזר דינמית** מ-`git remote` (לא מוקשח ל-`"mrtesy-app"`
  כמו היום ב-`build-session-proposal.mjs:121` ו-`post-session-summary.sh:92`),
  כך שאותם קבצים נשתלים בכל רפו ללא שינוי ומדווחים על הרפו הנכון.
- מותקן ב-`.claude/hooks/` + רישום `Stop` ב-`.claude/settings.json` בכל רפו.

### 2. שאלת הזהות בתחילת סשן — אכיפה כפולה
- **SessionStart hook** (מרחיב את `session-start.sh` הקיים): אם
  `CLAUDE_CODE_USER_EMAIL != chanoch@maor.org`, מזריק הנחיה
  (`additionalContext`) — "לפני כל דבר אחר: שאל 'מי אתה?', הצג את הרשימה,
  ורשום את הבחירה לקובץ-זהות של הסשן."
- **Stop hook** אוכף: אם זה חשבון non-manager **ואין** קובץ-זהות לסשן →
  חוסם (block) עם "לא זיהית את המשתמש — שאל עכשיו". אותו מנגנון אמין
  שמפעיל היום את הסיכום, לא בקשה רכה.
- **קובץ-הזהות** נכתב לספרייה זמנית של הסשן (scratchpad / `.claude/tmp`
  keyed ב-`CLAUDE_CODE_REMOTE_SESSION_ID`); ה-Stop hook קורא ממנו את
  האימייל של העובד ומשתמש בו כזהות לפתרון.

### 3. רשימת העובדים — טבלה בפלטפורמה (מומלץ)
ראה השוואה ב[שאלות פתוחות](#שאלות-פתוחות). ההמלצה: טבלה בפלטפורמה כי
הסביבה המרוחקת חולפת (קובץ לא-מקובץ לא שורד; קובץ מקובץ = churn מתמשך).
- טבלה חדשה, למשל `claude_known_workers` (org-scoped): `org_id`, `email`,
  `label`, `created_at`.
- שני routes תחת ה-namespace הקיים `claude-session` (מאובטח ב-`x-cron-secret`):
  - `GET /api/claude-session/known-workers` → הרשימה לתחילת הסשן.
  - `POST /api/claude-session/known-workers` → הוספת אימייל חדש.

### 4. מיפוי זהות → חשבון פלטפורמה
- **מנהל:** `chanoch@maor.org` (או בחירה מהרשימה) → `chanoch770@gmail.com`.
  ממומש כ-override קבוע ב-hook (כמו `SMRTTASK_USER_EMAIL` היום).
- **עובדים:** ✅ מאושר — האימייל שהעובד מקליד **הוא** אימייל ה-smrtTask שלו,
  נפתר ישירות בשרת (`resolveUserId` הקיים). אין מיפוי מיוחד לעובדים.

## שינויי שרת

1. **טבלה `claude_known_workers`** — migration חדש תחת
   `supabase/migrations/`.
2. **routes `known-workers`** (GET/POST) בתוך
   `server/src/modules/smrttask/routes/claude-session.ts` או קובץ אחות.
3. **"הצעה למנהל" בזרימת העובד** — הרחבת ה-endpoint כך שדיווח עובד יוצר
   גם הצעת smrtTask בתיבת המנהל (לצד עדכון משימת העובד). ניתן למחזר את
   לוגיקת ה-`proposal` הקיימת.

כל ה-endpoints נשארים machine-to-machine מאובטחים ב-`x-cron-secret`, אפס
טוקני API בתשלום (הסיכומים נכתבים ע"י הסוכן על המנוי).

## זרימת נתונים (תמצית)

```
SessionStart hook
  └─ CLAUDE_CODE_USER_EMAIL == chanoch@maor.org ?
        כן → זהות=מנהל (chanoch770@gmail.com); אין שאלה
        לא → הזרק הנחיה: שאל "מי אתה?" + הצג known-workers → כתוב קובץ-זהות

(הסוכן עובד על המשימה של המשתמש)

Stop hook (בסוף התור)
  ├─ non-manager ואין קובץ-זהות? → block ("שאל מי אתה")
  ├─ זהות = מנהל  → task-report; אם attached:false → proposal
  └─ זהות = עובד  → task-report (משימת העובד) + proposal (תיבת המנהל)
```

## שאלות פתוחות

1. **אחסון הרשימה** — מאושר לכיוון **טבלה בפלטפורמה** (המלצה)? או שמעדיף
   קובץ `.claude/known-workers.json` מקובץ למרות ה-churn?
2. **זיהוי עובדים** — ✅ מאושר: העובד מקליד את **אימייל ה-smrtTask שלו**,
   שנפתר ישירות. השאלה מנוסחת "מה האימייל שלך ב-smrtTask?" בעברית/אנגלית לפי
   שפת המשתמש. המיפוי המיוחד היחיד הוא של המנהל
   (`chanoch@maor.org`→`chanoch770@gmail.com`).
3. **הצעה למנהל** — ✅ מאושר: דיווח עובד → **הצעה** בתיבת המנהל (לא התראה).
4. **video-lab** — האם להשאיר גם את דיווח ה-smrtPlan הקיים (עם תוספת זהות),
   או שהזרימה החדשה מחליפה אותו לגמרי?
5. **עומס תיבת המנהל** — הצעה לכל סשן עובד עלולה להצטבר. רוצה dedup (הצעה
   אחת לכל עובד/יום, במקום לכל סשן)?

## מה לא משתנה

- אפס טוקני API בתשלום — כל הסיכומים נכתבים ע"י הסוכן על מנוי ה-Claude.
- כל ה-endpoints נשארים מאובטחים ב-`x-cron-secret`, ללא JWT.
- שינוי הזהות שכבר נפרס (`claude_user_email`/`claude_user_name`) נשאר ומשולב.
- הכל guarded: לא סשן web / חסר secret או URL / חסר `node`/`curl`/`jq` →
  no-op שקט שלא חוסם ולא מפיל תור.
