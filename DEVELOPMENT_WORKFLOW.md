# Development Workflow — Build, Verify, Proceed

## הבעיה שהתהליך הזה פותר

בפיתוח עם AI, שלוש בעיות חוזרות:
1. **פיצ'רים שתוכננו אבל לא נבנו** — סעיפים נשכחים או מדולגים
2. **פיצ'רים שקיימים אבל לא עובדים** — קוד נכתב בלי בדיקה בפועל
3. **תיקונים שנטען שבוצעו אבל לא באמת** — עריכת קוד בלי הוכחה שהבאג נפתר

## העקרונות

### עקרון 1: כל סעיף בתוכנית = משימה מנוטרת
- כל פיצ'ר מהתוכנית הופך ל-task ב-Todo List
- Task מסומן `in_progress` כשמתחילים לעבוד עליו
- Task מסומן `completed` **רק אחרי שאומת שהוא עובד**
- לא עוברים למשימה הבאה לפני שהנוכחית הושלמה ואומתה
- בסיום: ביקורת סופית — מעבר על התוכנית סעיף-סעיף מול מה שנבנה בפועל

### עקרון 2: כל פיצ'ר חייב אימות ויזואלי
אחרי כתיבת קוד לכל פיצ'ר, לפני שממשיכים:

```
1. הרצת TypeScript check (tsc --noEmit) → תיקון שגיאות טיפוסים
2. הפעלת שרת פיתוח (preview_start)
3. צילום מסך (preview_screenshot) → האם ה-UI נראה נכון?
4. בדיקת Console (preview_console_logs) → האם יש שגיאות JS?
5. בדיקת Network (preview_network) → האם קריאות API מצליחות?
6. קליק על אלמנטים (preview_click) → האם הניווט והאינטראקציות עובדים?
7. רק אז → מסמנים את המשימה כהושלמה
```

### עקרון 3: כל תיקון חייב הוכחה
כשמתקנים באג:

```
1. שחזור הבאג — צילום מסך / console שמראה את השגיאה
2. ביצוע התיקון בקוד
3. חזרה על אותם צעדים שגרמו לבאג
4. צילום מסך / console שמראה שהבעיה נפתרה
5. רק אז אומרים "תוקן"
```

**לעולם לא לומר "תיקנתי" בלי ראיה.**

### עקרון 4: פיתוח אינקרמנטלי
- בונים פיצ'ר **אחד** בכל פעם
- מאמתים שהוא עובד
- רק אז עוברים לפיצ'ר הבא
- זה מונע באגים מדורגים וקל יותר לבודד בעיות

## כלי האימות הזמינים

| כלי | מה הוא עושה | מתי להשתמש |
|------|-------------|-------------|
| `tsc --noEmit` | בדיקת טיפוסי TypeScript | אחרי כל שינוי בקוד |
| `preview_start` | הפעלת שרת פיתוח | לפני כל בדיקה ויזואלית |
| `preview_screenshot` | צילום מסך של העמוד | אימות UI |
| `preview_console_logs` | קריאת שגיאות Console | תפיסת שגיאות JS |
| `preview_network` | בדיקת קריאות API | אימות תקשורת עם שרת |
| `preview_click` | לחיצה על אלמנטים | בדיקת ניווט ואינטראקציות |
| `preview_fill` | מילוי טפסים | בדיקת form flows |
| `preview_inspect` | בדיקת CSS/styles | אימות עיצוב |
| `preview_snapshot` | עץ נגישות | אימות תוכן טקסטואלי |
| `preview_resize` | שינוי גודל viewport | בדיקת responsive |

## תבנית עבודה לכל פרויקט

```
שלב 1: תכנון
  └── כתיבת תוכנית מפורטת עם כל הפיצ'רים

שלב 2: המרה למשימות
  └── כל סעיף בתוכנית → task ב-Todo List

שלב 3: לולאת פיתוח (לכל פיצ'ר)
  ├── כתיבת קוד
  ├── tsc --noEmit
  ├── Preview → screenshot → console → network
  ├── בדיקת flow מלא
  ├── תיקון בעיות שנמצאו
  ├── אימות חוזר
  └── סימון task כהושלם

שלב 4: ביקורת סופית
  ├── מעבר על התוכנית סעיף-סעיף
  ├── בדיקה שכל דבר קיים ועובד
  └── דוח סטטוס סופי
```

## איך להשתמש במסמך הזה

הוסף את ההנחיה הבאה ל-CLAUDE.md של הפרויקט:

```markdown
## Development Rules
Follow the workflow in DEVELOPMENT_WORKFLOW.md:
- Every plan item must be a tracked task
- Every feature must be visually verified with Preview tools before marking complete
- Every bug fix requires proof (before/after screenshots or console output)
- Build one feature at a time — verify before moving on
```
