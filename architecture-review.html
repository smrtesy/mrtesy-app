סקירת ארכיטקטורה — התגובה להערה על Edge Functions

הערה מצוינת — והיא בערך 70% נכונה. אני אחדד איפה אני מסכים, איפה לא, ומה חסר בהצעה.

איפה הטענה צודקת (חזק)

Sync workers ארוכים — זו אכן החולשה הגדולה של Edge Functions.

  • Gmail full-scan של משתמש חדש עם 50K הודעות — קל לעבור את הגבול
  • Pagination + checkpoint דורשים לזכור state באמצע
  • אפשר לעקוף עם "re-invoke עצמי" אבל זה hack שבור בקלות

מסכים: לעבודה הזו — backend ייעודי בהחלט עדיף.

נקודה קטנה: הגבול של Supabase Edge הוא כיום 400s ולא 150s — אבל זה לא משנה את המסקנה, עדיין לא מספיק למשתמש חדש עם היסטוריה גדולה.

איפה הטענה חלשה

"AI pipeline סקוונציאלי לא שייך ל-Edge Function" — הטענה רופפת:

  • קריאה אחת ל-Claude על הודעה אחת = 3–10 שניות. זה מצוין ל-Edge.
  • daily_ai_budget_usd זו קריאה אחת ל-DB — לא "stateful logic"
  • דווקא לפעולות קצרות פר-משימה (drafting, classify של הודעה בודדת), Edge מנצח backend רגיל

"OAuth cold-start בעייתי" — זה קצת FUD:

  • Cold start של Edge Function הוא <500ms
  • Token refresh = קריאה אחת + כתיבה אחת ל-Vault, קצרצרה
  • ה"כשלים המבלבלים" לרוב נובעים מ-race conditions שיהיו גם ב-Node.js אם לא משתמשים ב-lock

מה באמת חסר בהצעה — הנקודה הכי חשובה

הדיון "Edge Functions vs Node backend" הוא הטעיה. הבחירה האמיתית היא:

יש לך Queue, או אין לך Queue?

המערכת הזו היא קלאסית event-driven background processing. הליבה שלה חייבת להיות:

User clicks "sync Gmail"
        ↓
    Enqueue job {user_id, source: gmail, type: full_sync}
        ↓
    Queue workers (Node.js) — idempotent, retryable, resumable
        ├── Fetch page 1 → write source_messages → save checkpoint
        ├── If > 50s elapsed → enqueue continuation job with cursor
        └── Trigger next stage: Enqueue {type: classify, source_message_id}
        ↓
    AI worker pulls from queue → Claude → write log_entries → maybe create task

בלי Queue — לא משנה אם זה Node או Edge, המערכת תישבר תחת scale.
עם Queue — אפשר לקרוא לזה איך שתרצה.

יתרונות של התור

  • אימפוטנטיות: עבודה יכולה להיכשל ולחזור בלי duplicate
  • Resumability: full-sync של 50K הודעות מתפצל אוטומטית
  • Budget enforcement: AI worker בודק budget לפני pull מהתור
  • מולטי-טננט מובנה: {user_id} בכל job, fair scheduling בין משתמשים
  • Observability: רואים בדיוק איפה משתמש תקוע

נקודות שחסרות בתוכנית של הצד השני — צריך להוסיף

  1. בחירת תור: BullMQ (Redis, בוגר) או PGMQ (Supabase native). המלצה — התחל עם PGMQ כדי להימנע מתשתית שלישית.
  2. Idempotency keys בכל job — חובה
  3. Dead letter queue — jobs שנכשלו 5 פעמים הולכים לצד
  4. Rate limiting פר-משתמש: Gmail API = 250 quota units/sec — token bucket
  5. Per-user credential isolation: tokens ב-Supabase Vault עם RLS, לא ב-ENV
  6. Observability: Sentry לשגיאות + dashboard פנימי שרואה תור פר-משתמש
  7. Webhook push מ-Gmail במקום polling — חוסך 90% מהעבודה אחרי הסינכרון הראשוני
  8. Scheduler ל-cron-per-user: לא cron גלובלי, אלא job שחוזר "re-check user X every 15 min"
